# CLAUDE.md + Skills 工作流审查设计

## 背景

当前仓库已经形成以 `CLAUDE.md` 为主入口、以 `.claude/skills/*` 为执行单元的 qa-flow 工作空间，并且已有一轮历史审计文档。下一轮工作不再是重复罗列旧问题，而是要在当前仓库现状上，重新对标 **Claude Code / `CLAUDE.md` / Skills** 生态的常见规范与设计理念，形成一份可确认、可执行的优化问题清单。

## 目标

本轮工作的目标是输出一份 **全仓工作流治理审查结果**，重点覆盖：

1. `CLAUDE.md` 与仓库级入口设计
2. Skills 的规范性、边界与引用关系
3. 配置、路径、rules、shared scripts 的 source-of-truth 治理
4. 文案、向导、选项、步骤顺序、用户体验与可读性
5. `CLAUDE.md + Skills + 多 agent` 协同方式的设计优缺点

本轮先做审查与确认，不直接进入仓库改造。

## 审查边界

- **范围**：全仓工作流治理
- **外部基线**：仅对标 Claude Code / `CLAUDE.md` / Skills 生态
- **输出偏好**：问题清单 + 优先级 + 改进建议 + 涉及文件
- **非目标**：本轮不直接改造文件，不扩展到通用 Agent 框架全面横评

## 审查方法

采用 **“基线对标 + 分轨审查 + 汇总归并”** 的方式执行。

### 方案选择理由

相比逐文件地毯式扫描，先建立外部基线可以避免问题定义过于主观；相比先只谈理念再落地，分轨审查可以更快把理论映射到仓库中的具体文件与 contract。

## 多 Agent 审查拆解

审查拆为 5 条并行轨道：

1. **生态基线轨**  
   调研 Claude Code / `CLAUDE.md` / Skills 生态中的常见规范、常见 contract 与常见反模式。

2. **主入口轨**  
   审核 `CLAUDE.md`、README、功能入口、技能索引、路由说明之间的一致性。

3. **Skill contract 轨**  
   审核 `.claude/skills/*/SKILL.md`、prompt、references、rules 之间的职责边界、引用方式与模板一致性。

4. **配置与路径轨**  
   审核 `.claude/config.json`、rules、shared scripts、路径引用、配置收口与 source-of-truth 一致性。

5. **体验与文案轨**  
   审核菜单、向导、步骤说明、选项设计、文案表达、可读性与心智负担。

每个 agent 只负责自己的轨道，避免单 agent 上下文过宽；最终由主控统一去重、归因和排序。

## 判定标准

每个问题统一按以下维度判断：

1. **规范性**：是否符合当前主流的 Skill / prompt / orchestration 设计习惯
2. **单一权威性**：同一规则、状态、路径、模板是否被多点维护
3. **可执行性**：文档、脚本、入口、测试是否真实可运行
4. **可读性**：长度、结构、职责边界、扫描成本是否合理
5. **可编排性**：步骤顺序、依赖关系、确认点、异常分支是否清楚
6. **可维护性**：后续新增 Skill / prompt / rule 时是否易漂移、漏改或失配

## 优先级模型

- **P0**：直接误导 agent 或阻断主流程
- **P1**：不会立刻失败，但持续制造漂移、重复维护或高误用率
- **P2**：偏文案、可读性、向导体验、模板润色、结构优化

## 交付格式

每个问题按统一模板输出：

- 问题描述
- 为什么是问题
- 影响范围
- 优先级
- 建议改法
- 涉及文件

必要时附加外部基线依据或仓库证据，但主交付物保持为一份可直接确认的优化问题清单。

## 执行顺序

1. 建立 Claude Code / `CLAUDE.md` / Skills 生态基线
2. 并行执行 5 条仓库内审查轨
3. 与现有审计文档交叉复核，避免重复上报已闭环问题
4. 汇总为当前仍需优化的问题清单
5. 交由用户确认后，再决定是否进入系统化整改

## 风险与控制

- **风险：历史问题重复上报**  
  通过与现有审计文档交叉复核控制。

- **风险：问题过碎、不可执行**  
  通过统一模板输出，并做根因归并控制。

- **风险：agent 结论重复或冲突**  
  由主控进行去重、归因与优先级收口。

## 完成定义

满足以下条件才算本轮审查完成：

1. 已建立外部基线
2. 已完成分轨审查
3. 已排除历史已闭环问题
4. 已形成当前仍需优化的问题清单
5. 问题清单已可供用户直接确认和后续排期
