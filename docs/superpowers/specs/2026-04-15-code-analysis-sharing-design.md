### 4 个场景截图对比（每个约 2 分钟）

| 场景 | 输入截图 | 输出截图 | 讲解重点 |
|------|---------|---------|---------|
| 1. 后端 Java 异常 | 一段 NPE 堆栈日志 | HTML 报告（含根因定位、调用链、修复建议） | 自动定位到源码行、区分环境问题 vs 代码问题 |
| 2. 前端 Console 错误 | 浏览器控制台 TypeError | HTML 报告（四层分析：组件/数据/环境/框架） | 自动识别错误类型并分层分析 |

### 第三步：使用 code-analysis（4 分钟）

4 种触发方式，各配一张操作截图:

| 触发方式 | 操作示例 | 说明 |
|---------|---------|------|
| 粘贴堆栈日志 | 直接把报错贴进终端 | 自动识别后端/前端 |
| 粘贴冲突代码 | 贴入含 `<<<<<<< HEAD` 的代码块 | 自动进入冲突分析 |

## 第三幕：揭秘拆解（10 分钟）

### 引入（30 秒）

- "刚才看到的效果，背后是怎么实现的？我们用 code-analysis 的工作流来拆解"

### 3.1 整体架构图（2 分钟）

用 drawio 绘制 code-analysis 全流程图:

```
用户输入 -> 模式识别(5种) -> 项目选择 -> 源码同步 -> Agent 分析 -> 模板渲染 -> 输出报告
```

引出: "这里面每一步都涉及一个 AI 概念，我们逐个拆解"

### 3.2 核心概念串讲（6 分钟）

按流程图节点顺序，逐个引出概念:

| 概念 | 一句话定义 | 在 code-analysis 中的体现 |
|------|-----------|-------------------------|
| Prompt（提示词） | 给 AI 的指令文本 | SKILL.md 里的模式识别规则、分析步骤 |
| 提示词工程 | 设计 prompt 使 AI 输出稳定可控 | 5 种模式的优先级判断、JSON 输出格式约束 |
| Token | AI 的计费和上下文单位 | 为什么用 Sonnet 而不是 Opus 跑 Agent（成本考量） |
| Agent | 能自主完成任务的 AI 子进程 | 4 个分析 Agent（后端/前端/冲突/Hotfix） |
| SubAgent | 被主流程调度的专用 Agent | code-analysis skill 调度 backend-bug-agent |
| Skill | 可复用的工作流模板 | `/code-analysis` 就是一个 skill |
| Hook | 事件触发的自动化动作 | 编辑后自动格式化、提交前安全检查 |
| MCP | AI 连接外部工具的协议 | Claude Code 通过 MCP 调用浏览器、Git 等外部工具 |

### 3.3 设计思想小结（1.5 分钟）

三个关键设计决策:

1. **模式路由** -- 不同输入走不同分析链路，而非一个万能 prompt
2. **双门禁** -- AI 不能悄悄改配置，每步需确认
3. **结构化输出** -- Agent 输出 JSON -> 模板渲染 HTML，保证格式稳定
