## 输入

任务提示中会指定以下路径：

- **原始用例数据路径**：从 `kata-cli history-convert` 输出的解析数据文件路径。读取该文件获取从 XMind/CSV 解析出的原始用例结构化数据。
- **偏好规则**：读取 `rules/` 目录下的所有规则文件。

同时读取：

- `.claude/skills/test-case-gen/references/test-case-standards.md` — 用例编写规范（权威来源）
- `.claude/skills/test-case-gen/references/intermediate-format.md` — 中间 JSON 格式规范

## 步骤

### 步骤 1：读取原始用例数据

从任务提示指定的路径读取 history-convert.ts 输出的解析数据，理解原始用例结构：

- 用例标题（可能是裸标题，也可能是 `【P1】验证xxx` 这类展示标题）
- 步骤描述（可能模糊、缺少三要素）
- 预期结果（可能不可断言）
- 前置条件（可能缺失或笼统）

### 步骤 2：读取偏好规则

读取 `rules/` 目录下的规则文件，了解项目特定的：

- 菜单结构和导航路径
- 业务术语和字段命名
- 数据源类型偏好

### 步骤 3：逐条标准化重写

对每条原始用例执行以下标准化操作：

#### 标题标准化

- 根据用例重要性判定 `priority=P0/P1/P2`
  - P0：核心业务流程、数据完整性、权限控制
  - P1：常规功能验证、表单校验、列表操作
  - P2：边界值、兼容性、UI 细节
- 中间 JSON / XMind 标题统一为裸标题 `验证xxx`
- 最终 Archive MD / 展示渲染时，再组合为 `【P0/P1/P2】验证xxx`
- 原始标题中的信息保留但重新措辞

#### 步骤标准化（三要素强制）

将每个原始步骤重写为包含：

1. **操作位置**：明确控件所在区域（页面顶部搜索栏 / 表单区 / 表格操作列 / 弹窗内）
2. **操作对象**：字段名 + 控件类型（「商品名称」输入框 / 「状态」下拉框 / 【提交】按钮）
3. **操作动作 + 具体值**：输入"2026春季新款运动鞋" / 选择"上架" / 点击

**第一步**必须以「进入【导航路径】页面，等待xxx加载完成」开头。

**等待条件**：导航/搜索/提交/弹窗操作后必须注明等待条件。

**重写示例**：

```
原始：搜索商品
重写：在页面顶部搜索栏的「商品名称」输入框中输入"2026春季新款运动鞋"，点击输入框右侧的【搜索】按钮，等待列表刷新完成

原始：新增一条数据
重写：点击列表页右上角的【新增】按钮，等待新增弹窗完全展开

原始：填写表单并提交
重写：在【新增商品】弹窗中按顺序配置如下：
- 商品名称: 2026春季新款运动鞋
- 商品分类: 运动鞋（下拉框选择）
- 价格: 299.00
- 状态: 上架（下拉框选择）
点击弹窗底部的【确定】按钮，等待提交请求完成
```

#### 预期结果标准化（可断言化）

将模糊预期重写为 Playwright AI 可直接翻译为 `expect()` 断言的表述：

```
原始：操作成功
重写：页面顶部弹出绿色提示"新增成功"（约 3 秒后自动消失）；列表自动刷新，第一行显示新增记录，「商品名称」列为"2026春季新款运动鞋"

原始：数据正确
重写：列表第一行各列显示：商品名称"2026春季新款运动鞋"，分类"运动鞋"，价格"299.00"，状态"上架"

原始：校验失败
重写：「商品名称」输入框下方显示红色提示文字"请输入商品名称"；【确定】按钮保持禁用状态（灰色不可点击）
```

#### 前置条件标准化（操作化）

将模糊前置条件重写为具体的操作步骤或 SQL：

```
原始：系统中有测试数据
重写：
1、使用账号 admin账号 登录系统
2、在【商品管理 → 商品列表】页面已通过新增功能创建以下商品：
   - 商品名称"2026春季新款运动鞋"，分类"运动鞋"，价格 299.00，状态"上架"
   - 商品名称"商务休闲皮鞋"，分类"皮鞋"，价格 599.00，状态"下架"

原始：（无前置条件）
重写：使用 admin账号 登录系统
```

#### 用例合并与拆分

- 步骤数 < 4 的正向用例，合并到同功能区的综合正向用例中
- 一条用例测试多个逆向条件的，拆分为每条只测一个逆向条件
- 表单多字段输入，合并为一个步骤（按渲染顺序列出）

#### 占位数据替换

将 test1、abc、123456 等占位符替换为贴近真实业务的数据：

| 占位数据            | 替换为                                 |
| ------------------- | -------------------------------------- |
| test / test1 / 测试 | 具体业务名称（如"2026春季新款运动鞋"） |
| abc / xxx           | 具体编码（如"SKU-2026-001"）           |
| 123456              | 具体数值（如"299.00"）                 |
| admin               | 保留（登录账号通常就是 admin）         |

## 输出格式

JSON 结构参见 `docs/architecture/references/output-schemas.json` 中的 `standardize_json`；输出结构同时遵循 `.claude/skills/test-case-gen/references/intermediate-format.md`，标题字段遵循 Contract A。

## 输出

写出标准化 JSON 后，打印如下摘要：

```
用例标准化完成
  源文件:          <原始文件路径>
  原始用例数:      <N> 条
  标准化后用例数:  <N> 条（合并 <N> 条，拆分 <N> 条）
  优先级分布:      P0: <N> / P1: <N> / P2: <N>
  待确认项:        <N> 条
```

## 错误处理

遵循 `docs/architecture/references/error-handling-patterns.md` 中的标准分类与恢复策略。

## 重写原则

1. **保留原始测试意图**：不删减原始用例覆盖的测试点，只提升描述精度
2. **补充缺失信息**：原始用例缺少的前置条件、等待条件、具体数据，根据上下文合理补充
3. **无法推断时标注**：如果原始用例的业务上下文不足以推断具体值，在该字段标注 `[待确认：xxx]`，后续触发用户确认
4. **不添加原始未覆盖的测试点**：标准化是重写，不是新增
