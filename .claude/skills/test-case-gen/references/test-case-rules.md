# 测试用例编写硬性规则

> 本文件是用例编写的唯一权威来源。Writer Subagent 和 Reviewer Subagent 必须严格遵守。

---

## 1. XMind 层级结构

```
Root（项目名）
  └── L1：需求名称（prd_id 写入 L1 labels）          ← trackerId 模块，L1 不加版本前缀
  └── L1：【版本】需求名称（prd_id 写入 L1 labels）   ← 无 trackerId 模块，L1 带版本前缀
       └── L2：模块 / 菜单名称       ← 如「商品管理」
            └── L3：页面名称          ← 如「列表页」「新增页」「详情页」
                 └── [L4：功能子组]   ← 如「搜索」「字段校验」（可选）
                      └── 用例标题
                           └── 步骤
                                └── 预期结果
```

- L1 格式由 `config.json` 中模块是否配置 `trackerId` 决定
- L2 必须对应系统中的真实菜单或独立功能模块名称
- L3 按页面维度划分（列表页、新增页、编辑页、详情页等）
- L4 为可选层级，当页面内部功能较多时按功能分组使用

---

## 2. Archive MD 层级结构

```
（frontmatter: suite_name = 需求名称）
## 模块名称         ← 对应 XMind L2
### 页面名称        ← 对应 XMind L3
#### 功能子组       ← 对应 XMind L4（可选）
##### 【P0】验证xxx ← 用例标题（含优先级前缀）
```

- Archive MD body 从 `##` 开始，**无** `#`（H1）标题
- 需求名称统一存入 frontmatter `suite_name` 字段

---

## 3. 用例标题规范

- **优先级前缀**：`【P0】`、`【P1】`、`【P2】`，置于用例标题**最前面**
- 「验证」二字紧跟优先级标记，例如：`【P1】验证填写完整表单后成功提交`
- 标题应概括测试目的，**不包含**具体操作步骤
- 禁止在优先级括号内外多余的标点（如 `【P1】:` 或 `[P1]`）

---

## 4. 步骤格式规范

### 4.1 禁止步骤编号前缀

不允许出现「步骤1:」「步骤2:」「Step1:」「S1.」等任何编号前缀。

### 4.2 第一步必须是导航步骤

第一个步骤**必须**以「进入【完整导航路径】页面」开头：

| 情形     | 正确写法                          | 禁止写法                                                |
| -------- | --------------------------------- | ------------------------------------------------------- |
| 单层菜单 | `进入【商品列表】页面`            | `打开商品列表`                                          |
| 多层菜单 | `进入【商品管理 → 商品列表】页面` | `进入【商品列表】页面（导航路径：商品管理 → 商品列表）` |

- 多层菜单路径放在 `【】` 内，各层级用 `→` 连接
- **禁止**将导航路径放入括号注释中

### 4.3 步骤表格格式

步骤表格前**必须**有 `> 用例步骤` 标签：

```markdown
> 用例步骤

| 编号 | 步骤                            | 预期               |
| ---- | ------------------------------- | ------------------ |
| 1    | 进入【商品管理 → 商品列表】页面 | 页面正常加载       |
| 2    | 点击【新增】按钮                | 弹出新增商品对话框 |
```

## 5. 用例设计原则

| 类型     | 条件组合规则           | 示例                                              |
| -------- | ---------------------- | ------------------------------------------------- |
| 正向用例 | 可包含多个正向条件组合 | 验证填写完整表单（名称+分类+价格+库存）后成功提交 |
| 逆向用例 | **仅允许一个逆向条件** | 验证商品名称为空时无法提交                        |
| 边界用例 | 针对单一边界值验证     | 验证商品名称输入 100 个字符时正常保存             |

### 5.1 正向用例合并规则（强制）

同一功能区（sub_group 或 page）内，步骤数少于 4 条的正向用例**必须**与其他正向用例合并：

- 将该功能区内所有步骤数 `< 4` 的正向用例整合为 1 个综合正向用例
- 综合用例步骤按用户操作流顺序衔接，前置条件取各子用例的并集
- **异常用例不适用此规则**：每条异常用例仍只测试一个逆向条件，步骤数即使只有 1 条也不合并

## 6. 测试数据规范

**必须使用真实业务数据**，给出具体的输入值和选择项。

### 6.1 禁止词对照表

| 禁止词（模糊写法）         | 正确写法（具体数据）                                               |
| -------------------------- | ------------------------------------------------------------------ |
| 尝试提交表单               | 点击【提交】按钮                                                   |
| 输入像"测试数据"这样的内容 | 在「商品名称」输入框输入"2026春季新款运动鞋"                       |
| 选择如"已完成"等状态       | 在「状态」下拉框选择"上架"                                         |
| 输入比如手机号             | 在「联系电话」输入框输入"13800138000"                              |
| 填写相关信息               | 在「商品名称」输入"2026春季新款运动鞋"，在「商品分类」选择"运动鞋" |
| 输入合法的数值             | 在「价格」输入框输入"299.00"                                       |
| 选择某个日期               | 在「生效日期」选择"2026-04-01"                                     |

## 7. 预期结果规范

必须**精确描述系统行为**，包含可观测的具体表现。

### 7.1 禁止与正确写法对照表

| 禁止写法（空洞表述） | 正确写法（精确可观测）                                                       |
| -------------------- | ---------------------------------------------------------------------------- |
| 操作成功             | 页面提示"新增成功"，列表新增一条记录，商品名称显示"2026春季新款运动鞋"       |
| 显示正确             | 列表第一行显示：商品名称"2026春季新款运动鞋"，分类"运动鞋"，状态"上架"       |
| 提交失败             | 「商品名称」输入框下方显示红色提示"请输入商品名称"，【提交】按钮保持不可点击 |
| 页面跳转             | 跳转到「商品详情」页面，页面标题显示"2026春季新款运动鞋"                     |
| 数据更新             | 列表中对应记录的状态列由"下架"变更为"上架"                                   |

## 8. 表单多字段步骤合并规则（强制）

当一个步骤涉及同一表单 / 面板内的多个配置项时，**必须**合并为一个步骤节点，按字段在页面上的实际渲染顺序逐项列出。

### 8.1 正确写法

```text
在【新增商品】表单页面，点击【保存】前按顺序配置如下：
- 商品名称: 2026春季新款运动鞋
- 商品分类: 运动鞋
- SKU编号: SKU-2026-001
- 价格: 299.00
- 库存数量: 100
- 状态: 上架
- 商品卖点: 轻量缓震跑鞋（非必填，可填写无）
```

### 8.2 禁止写法（每个字段单独一个步骤）

```text
❌ 步骤1: 在「商品分类」下拉框选择「运动鞋」
❌ 步骤2: 在「商品名称」输入框输入「2026春季新款运动鞋」
❌ 步骤3: 在「价格」输入框输入「299.00」
```

### 8.3 适用规则

1. 字段顺序：按前端页面实际渲染顺序排列，优先参考前端源码
2. 非必填字段：保留在列表中，在值后加注 `（非必填，可填写无）`
3. 嵌套配置：若某字段展开后有子配置项，用缩进或换行继续列举
4. **拆分例外**：以下情况允许拆分为单独步骤：
   - 该字段选择会触发页面联动 / 刷新，且联动后需单独验证
   - 该步骤本身是该用例的核心验证动作

## 9. 前置条件数据准备规则

**当需求涉及数据表操作、数据接入、批量同步、任务调度、对账、规则配置、血缘解析、指标生成、历史升级 / 补数 / 回刷等能力时，必须给出具体可执行的前置条件。**

### 9.1 格式示例

```text
1、环境说明：目标模块已部署，数据源连接正常

2、Doris2.x SQL 语句准备:
DROP TABLE IF EXISTS qa_test.table_a;
CREATE TABLE qa_test.table_a (
  id INT,
  name VARCHAR(100),
  value DECIMAL(10,2),
  status VARCHAR(20),
  create_time DATETIME
);
INSERT INTO qa_test.table_a VALUES
  (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');

3、Hive2.x SQL 语句准备:
CREATE TABLE IF NOT EXISTS qa_test.table_a (
  id INT,
  name STRING,
  value DECIMAL(10,2),
  status STRING,
  create_time TIMESTAMP
);
INSERT INTO qa_test.table_a VALUES (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');
```

### 9.2 血缘 / 同步 / 回刷类场景额外约束

出现以下关键词时，前置条件还必须包含「链路如何建立」：

- `血缘`、`上下游`、`解析`
- `脚本模式指标`、`向导模式指标`
- `同步`、`调度任务`
- `历史数据`、`升级`、`补数`、`回刷`

此类场景前置条件至少包含：

1. **执行目标**：在哪个引擎 / 集群执行；Doris2.x 和 Doris3.x 复用相同库表名时必须明确集群
2. **建表 / 灌数 SQL**：真实表结构与真实业务数据
3. **建链 SQL / 任务 SQL**：如 `INSERT OVERWRITE`、调度任务 SQL、解析链路所依赖的实际 SQL
4. **指标定义 / 脚本 SQL**：若涉及指标，给出原子 / 派生 / 复合指标来源，或完整脚本模式 SQL
5. **历史回刷 SQL**：若验收点涉及历史兼容，给出历史样例准备方式或回刷 SQL
6. **执行顺序**：若结果依赖操作顺序，必须写明先后关系

**禁止写法：**

- `确保它们与 xxx 建立可追溯血缘关系`
- `准备历史指标样例`
- `同步相关数据后即可验证`

## 10. 质量门禁

Reviewer 采用 3 级质量阈值自动决策：

| 问题率      | 行为                                 |
| ----------- | ------------------------------------ |
| `< 15%`     | 静默自动修正，不通知用户             |
| `15% - 40%` | 自动修正 + 质量警告，建议用户核查    |
| `> 40%`     | 立即停止，输出阻断报告，等待用户决策 |

**问题率计算公式**：含 F01–F18 任意问题的用例数 ÷ 总用例数（一条用例多个问题只计 1 次）

## 11. 自动化可执行精度（Playwright AI 友好）

> 本项目生成的用例需同时满足**人类可读**和 **Playwright AI 可直接执行**两个目标。每条用例的步骤、预期、前置条件都必须达到"AI 不需要猜测即可翻译为自动化脚本"的精度。

### 11.1 步骤精度要求

每个步骤必须包含以下三要素，缺一不可：

| 要素 | 说明 | 正确示例 | 禁止示例 |
| --- | --- | --- | --- |
| **操作对象** | 字段名 + 控件类型（输入框/下拉框/按钮/复选框/标签页） | 在「商品名称」输入框中 | 在输入框中 |
| **操作动作** | 输入/选择/点击/勾选/切换/拖拽 + 具体值 | 输入"2026春季新款运动鞋" | 输入商品名称 |
| **操作位置** | 控件所在的页面区域（顶部搜索栏/表单区/表格操作列/弹窗内） | 在页面顶部搜索栏的「商品名称」输入框 | 在搜索框 |

**导航步骤强化**：

```
❌ 进入【商品管理 → 商品列表】页面
✅ 进入【商品管理 → 商品列表】页面，等待列表数据加载完成（表格至少显示 1 行数据或空态提示）
```

**等待条件**：涉及异步加载的步骤（页面跳转、列表刷新、弹窗打开）必须在步骤末尾注明等待条件：
- 列表加载：`等待表格数据加载完成`
- 弹窗：`等待弹窗完全展开`
- 页面跳转：`等待页面 URL 变为 /xxx 或页面标题变为「xxx」`
- 提交后：`等待提交请求完成，页面跳转回列表页`

### 11.2 预期结果精度要求

预期结果必须是**可断言的具体表现**，包含以下要素：

| 要素 | 说明 | 正确示例 | 禁止示例 |
| --- | --- | --- | --- |
| **可见文本** | 精确到页面上可观测的文字 | 页面提示"新增成功" | 操作成功 |
| **元素状态** | 可见/不可见/禁用/选中/高亮 | 【提交】按钮变为禁用（灰色不可点击） | 按钮不可用 |
| **数据变化** | 具体的前后对比 | 列表总数从"共 50 条"变为"共 51 条"，新增记录排在第一行 | 列表更新 |
| **位置/排序** | 元素出现的位置 | 列表第一行的「商品名称」列显示"2026春季新款运动鞋" | 列表显示新记录 |

**复合预期**：当一个步骤有多个可观测结果时，用分号分隔逐一列出：

```
✅ 列表刷新，仅显示商品名称包含"运动鞋"的记录；列表左上角总数从"共 50 条"变为"共 3 条"；搜索框保留输入值"运动鞋"
❌ 搜索结果正确显示
```

### 11.3 前置条件精度要求

前置条件不能只描述"需要什么状态"，必须描述**如何达到该状态**：

| 场景 | 禁止写法 | 正确写法 |
| --- | --- | --- |
| 需要已有数据 | `系统中已有商品数据` | `在「商品管理 → 商品列表」页面，已存在至少 3 条商品记录（可通过新增页面手动创建，或执行以下 SQL 插入）` |
| 需要特定状态 | `商品处于上架状态` | `通过【商品列表】页面的「操作」列点击「上架」按钮，将商品"2026春季新款运动鞋"的状态变更为"上架"` |
| 需要登录态 | `已登录系统` | `使用账号 admin / 密码 Admin@123 登录系统（登录地址：系统首页登录表单）` |
| 需要配置项 | `已配置数据源` | `在【系统管理 → 数据源管理】页面，已新增 Doris 类型数据源，连接信息：host=10.0.0.1, port=9030, database=qa_test` |

**数据库前置条件**：若涉及数据表操作，必须给出完整可执行 SQL（见第 9 节），且额外注明：
- 在哪个数据库/集群执行
- SQL 执行顺序
- 执行后的预期数据状态（如"执行后 qa_test.table_a 应有 3 条记录"）

### 11.4 整体示例（自动化可执行级别）

```markdown
##### 【P1】验证按商品名称搜索返回正确结果

> 前置条件
```
1、使用账号 admin / 密码 Admin@123 登录系统
2、在【商品管理 → 商品列表】页面，已存在以下商品记录：
   - 商品名称"2026春季新款运动鞋"，分类"运动鞋"，状态"上架"
   - 商品名称"2026夏季透气跑步鞋"，分类"运动鞋"，状态"上架"
   - 商品名称"商务休闲皮鞋"，分类"皮鞋"，状态"上架"
```

> 用例步骤

| 编号 | 步骤 | 预期 |
| --- | --- | --- |
| 1 | 进入【商品管理 → 商品列表】页面，等待列表数据加载完成（表格显示 3 条记录） | 页面正常加载，列表左上角显示"共 3 条"，表格展示 3 行商品数据 |
| 2 | 在页面顶部搜索栏的「商品名称」输入框中输入"运动鞋" | 输入框显示文本"运动鞋" |
| 3 | 点击搜索栏右侧的【搜索】按钮，等待列表刷新完成 | 列表刷新，仅显示 2 条记录；列表左上角总数变为"共 2 条"；两条记录分别为"2026春季新款运动鞋"和"2026夏季透气跑步鞋"；"商务休闲皮鞋"不在结果中 |
| 4 | 点击搜索栏右侧的【重置】按钮，等待列表刷新 | 搜索输入框清空；列表恢复显示全部 3 条记录；左上角总数恢复为"共 3 条" |
```
