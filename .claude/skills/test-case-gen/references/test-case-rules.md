## 1. XMind 层级结构

```
Root（项目名）
  └── L1：需求名称（prd_id 写入 L1 labels）          ← trackerId 模块，L1 不加版本前缀
  └── L1：【版本】需求名称（prd_id 写入 L1 labels）   ← 无 trackerId 模块，L1 带版本前缀
       └── L2：模块名称             ← 如「数据质量」「商品管理」
            └── L3：菜单名称        ← 如「json格式校验管理」「商品列表」（二级或三级菜单用横杠连接，如「通用配置-json格式校验管理」）
                 └── [L4：功能点]   ← 如「新增key」「搜索筛选」（可选）
                      └── L5：用例标题（title=验证xxx，priority 单独存储）
                           └── 步骤
                                └── 预期结果
```

- L1 格式由 `config.json` 中模块是否配置 `trackerId` 决定
- L2 必须对应系统中的真实模块名称（如「数据质量」「数据安全」）
- **L3 是菜单名称**（如「json格式校验管理」），**不要**附加「-列表页」「-新增页」等页面后缀。如果模块下有多级菜单，通过横杠连接（如 `通用配置-json格式校验管理`）
- L4 为可选层级，当菜单内部功能较多时按功能点分组使用（如「新增key」「编辑key」「删除key」）

---

## 2. Archive MD 层级结构

```
（frontmatter: suite_name = 需求名称，对应 XMind L1 节点）
## 模块名称          ← 对应 XMind L2（如「数据质量」）
### 菜单名称         ← 对应 XMind L3（如「json格式校验管理」，多级菜单用横杠连接）
#### 功能点           ← 对应 XMind L4（可选，如「新增key」）
##### 【P0】验证xxx  ← Archive MD / display 标题；对应同一条 XMind L5 用例
```

- Archive MD body 从 `##` 开始，**无** `#`（H1）标题
- 需求名称统一存入 frontmatter `suite_name` 字段
- `suite_name` 对应 XMind 的 L1 节点名称
- H2 对应 L2（模块），H3 对应 L3（菜单），H4 对应 L4（功能点），H5 对应同一条 L5 用例的 Archive / display 标题
- Archive MD 的 H5 使用 Contract B；XMind L5 节点自身保持 Contract A 裸标题，优先级通过 marker / metadata 表达

## 3. Archive MD / 展示标题规范

- **优先级前缀**：`【P0】`、`【P1】`、`【P2】`，置于用例标题**最前面**
- 「验证」二字紧跟优先级标记，例如：`【P1】验证填写完整表单后成功提交`
- 标题应概括测试目的，**不包含**具体操作步骤
- 禁止在优先级括号内外多余的标点（如 `【P1】:` 或 `[P1]`）

## 4. 步骤格式规范

### 4.1 禁止步骤编号前缀

不允许出现「步骤1:」「步骤2:」「Step1:」「S1.」等任何编号前缀。

### 4.2 第一步必须是导航步骤

第一个步骤**必须**以「进入【完整导航路径】页面」开头，且注明等待条件：

| 情形     | 正确写法                                                               | 禁止写法                                                |
| -------- | ---------------------------------------------------------------------- | ------------------------------------------------------- |
| 单层菜单 | `进入【商品列表】页面，等待列表数据加载完成`                           | `打开商品列表`                                          |
| 多层菜单 | `进入【商品管理 → 商品列表】页面，等待表格至少显示 1 行数据或空态提示` | `进入【商品列表】页面（导航路径：商品管理 → 商品列表）` |

- 多层菜单路径放在 `【】` 内，各层级用 `→` 连接
- **禁止**将导航路径放入括号注释中

### 4.4 步骤三要素（自动化精度）

> 本项目用例将被 Playwright AI 直接转化为自动化脚本，步骤精度不足会导致自动化失败。

每个步骤必须包含以下三要素，缺一不可：

| 要素                  | 说明              | 正确示例                 | 禁止示例     |
| --------------------- | ----------------- | ------------------------ | ------------ |
| **操作位置**          | 控件所在区域      | 在页面顶部搜索栏的       | 在搜索框     |
| **操作对象**          | 字段名 + 控件类型 | 「商品名称」输入框       | 输入框       |
| **操作动作 + 具体值** | 动词 + 精确值     | 输入"2026春季新款运动鞋" | 输入商品名称 |

### 4.5 等待条件（异步操作后必须注明）

| 场景      | 等待条件写法                                        |
| --------- | --------------------------------------------------- |
| 导航/跳转 | `等待列表数据加载完成` 或 `等待页面标题变为「xxx」` |
| 搜索/筛选 | `等待列表刷新完成`                                  |
| 表单提交  | `等待提交请求完成，页面跳转回列表页`                |
| 弹窗打开  | `等待弹窗完全展开`                                  |

### 4.3 预期结果不允许留空

**每个步骤必须有对应的预期结果，预期列禁止留空。** 即使步骤是中间操作（如导航、填写表单），也必须填写预期结果以确认该步骤执行成功。

常见中间步骤的预期结果写法：

| 步骤类型             | 示例步骤                           | 预期结果写法                                 |
| -------------------- | ---------------------------------- | -------------------------------------------- |
| 页面导航             | 进入【数据质量 → 规则集管理】页面  | 规则集管理页面正常打开，列表加载完成，无报错 |
| 列表搜索/筛选        | 在搜索框中输入关键字               | 列表根据关键字过滤，匹配结果正确展示         |
| 表单填写（中间步骤） | 在规则配置表单中按顺序填写各项     | 各表单项可正常输入/选择，无异常提示          |
| 点击编辑进入表单     | 点击操作列的【编辑】按钮           | 表单页正常打开，已有配置项回显正确           |
| 弹窗/对话框打开      | 点击【查看详情】链接               | 详情弹窗正常打开，数据加载完成               |
| Tab 切换             | 点击【自定义规则】Tab              | Tab 切换成功，对应内容区正常加载             |
| 下拉选择             | 在「选择数据源」下拉框选择某数据源 | 下拉框显示所选值，关联字段联动刷新           |

### 4.6 步骤表格格式

步骤表格前**必须**有 `> 用例步骤` 标签：

```markdown
> 用例步骤

| 编号 | 步骤                            | 预期               |
| ---- | ------------------------------- | ------------------ |
| 1    | 进入【商品管理 → 商品列表】页面 | 页面正常加载       |
| 2    | 点击【新增】按钮                | 弹出新增商品对话框 |
```

## 5. 用例设计原则

| 类型     | 条件组合规则           | 示例                                              |
| -------- | ---------------------- | ------------------------------------------------- |
| 正向用例 | 可包含多个正向条件组合 | 验证填写完整表单（名称+分类+价格+库存）后成功提交 |
| 逆向用例 | **仅允许一个逆向条件** | 验证商品名称为空时无法提交                        |
| 边界用例 | 针对单一边界值验证     | 验证商品名称输入 100 个字符时正常保存             |

### 5.1 正向用例合并规则（强制）

同一功能区（sub_group 或 page）内，步骤数少于 4 条的正向用例**必须**与其他正向用例合并：

- 将该功能区内所有步骤数 `< 4` 的正向用例整合为 1 个综合正向用例
- 综合用例步骤按用户操作流顺序衔接，前置条件取各子用例的并集
- **异常用例不适用此规则**：每条异常用例仍只测试一个逆向条件，步骤数即使只有 1 条也不合并

## 7. 预期结果规范

预期结果必须是 **Playwright AI 可直接翻译为 `expect()` 断言** 的可观测表现。

### 7.1 断言维度

| 断言维度  | 写法要求            | 正确示例                                               | 禁止示例   |
| --------- | ------------------- | ------------------------------------------------------ | ---------- |
| 可见文本  | 精确到页面上的文字  | 页面提示"新增成功"                                     | 操作成功   |
| 元素状态  | 可见/禁用/选中/高亮 | 【提交】按钮变为禁用状态（灰色不可点击）               | 按钮不可用 |
| 数据变化  | 具体前后对比        | 列表总数从"共 50 条"变为"共 51 条"，新增记录排在第一行 | 列表更新   |
| 位置/排序 | 元素出现的具体位置  | 列表第一行「商品名称」列显示"2026春季新款运动鞋"       | 新记录出现 |

### 7.2 禁止与正确写法对照表

| 禁止写法（空洞表述） | 正确写法（可断言）                                                           |
| -------------------- | ---------------------------------------------------------------------------- |
| 操作成功             | 页面提示"新增成功"，列表新增一条记录，商品名称显示"2026春季新款运动鞋"       |
| 显示正确             | 列表第一行显示：商品名称"2026春季新款运动鞋"，分类"运动鞋"，状态"上架"       |
| 提交失败             | 「商品名称」输入框下方显示红色提示"请输入商品名称"，【提交】按钮保持不可点击 |
| 页面跳转             | 跳转到「商品详情」页面，页面标题显示"2026春季新款运动鞋"                     |
| 数据更新             | 列表中对应记录的状态列由"下架"变更为"上架"                                   |
| 搜索成功             | 列表仅显示名称含"运动鞋"的记录；左上角总数变为"共 3 条"；搜索框保留"运动鞋"  |

### 7.3 复合预期

当一个步骤有多个可观测结果时，用分号分隔逐一列出：

```
✅ 列表刷新，仅显示 2 条记录；左上角总数变为"共 2 条"；"商务休闲皮鞋"不在结果中
❌ 搜索结果正确
```

## 8. 表单多字段步骤合并规则（强制）

当一个步骤涉及同一表单 / 面板内的多个配置项时，**必须**合并为一个步骤节点，按字段在页面上的实际渲染顺序逐项列出。**在 JSON 输出中，必须使用 `\n` 换行符 + `- 字段: 值` 列表格式，禁止将所有字段用逗号/顿号堆在一行。**

### 8.1 正确写法

渲染后效果：

```text
在【新增商品】表单页面，点击【保存】前按顺序配置如下：
- 商品名称: 2026春季新款运动鞋
- 商品分类: 运动鞋
- SKU编号: SKU-2026-001
- 价格: 299.00
- 库存数量: 100
- 状态: 上架
- 商品卖点: 轻量缓震跑鞋（非必填，可填写无）
```

对应的 JSON `step` 字段写法（**注意 `\n` 换行**）：

```json
{
  "step": "在【新增商品】表单页面，点击【保存】前按顺序配置如下：\n- 商品名称: 2026春季新款运动鞋\n- 商品分类: 运动鞋\n- SKU编号: SKU-2026-001\n- 价格: 299.00\n- 库存数量: 100\n- 状态: 上架\n- 商品卖点: 轻量缓震跑鞋（非必填，可填写无）",
  "expected": "各表单项可正常输入/选择，无异常提示"
}
```

### 8.2 禁止写法 1 — 每个字段单独一个步骤

```text
❌ 步骤1: 在「商品分类」下拉框选择「运动鞋」
❌ 步骤2: 在「商品名称」输入框输入「2026春季新款运动鞋」
❌ 步骤3: 在「价格」输入框输入「299.00」
```

### 8.3 禁止写法 2 — 所有字段堆在一行（逗号/顿号连接）

```json
{
  "step": "❌ 在【新增商品】表单页面填写商品名称为\"2026春季新款运动鞋\"、商品分类为\"运动鞋\"、SKU编号为\"SKU-2026-001\"、价格为\"299.00\"、库存数量为\"100\"、状态为\"上架\"，然后点击【保存】按钮"
}
```

### 8.4 适用规则

1. 字段顺序：按前端页面实际渲染顺序排列，优先参考前端源码
2. 非必填字段：保留在列表中，在值后加注 `（非必填，可填写无）`
3. 嵌套配置：若某字段展开后有子配置项，用缩进或换行继续列举
4. **拆分例外**：以下情况允许拆分为单独步骤：
   - 该字段选择会触发页面联动 / 刷新，且联动后需单独验证
   - 该步骤本身是该用例的核心验证动作

## 9. 前置条件数据准备规则

前置条件必须描述**如何达到该状态**，而非只描述"需要什么状态"。Playwright AI 会按前置条件准备测试环境，模糊的前置条件会导致自动化无法执行。

### 9.0 通用前置条件精度

| 禁止写法           | 正确写法                                                                                               |
| ------------------ | ------------------------------------------------------------------------------------------------------ |
| 系统中已有商品数据 | 已通过【商品管理→商品列表→新增】创建商品"2026春季新款运动鞋"（分类: 运动鞋, 价格: 299.00, 状态: 上架） |
| 已登录系统         | 使用 admin 账号登录系统                                                                                |
| 已配置数据源       | 在【系统管理→数据源管理】已新增 Doris 数据源（host=10.0.0.1, port=9030, database=qa_test）             |
| 商品处于上架状态   | 通过列表页操作列点击「上架」，将"2026春季新款运动鞋"状态变为"上架"                                     |

**当需求涉及数据表操作、数据接入、批量同步、任务调度、对账、规则配置、血缘解析、指标生成、历史升级 / 补数 / 回刷等能力时，还须给出具体可执行的 SQL 前置条件。**

### 9.1 格式示例

```text
1、环境说明：目标模块已部署，数据源连接正常

2、Doris2.x SQL 语句准备:
DROP TABLE IF EXISTS qa_test.table_a;
CREATE TABLE qa_test.table_a (
  id INT,
  name VARCHAR(100),
  value DECIMAL(10,2),
  status VARCHAR(20),
  create_time DATETIME
);
INSERT INTO qa_test.table_a VALUES
  (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');

3、Hive2.x SQL 语句准备:
CREATE TABLE IF NOT EXISTS qa_test.table_a (
  id INT,
  name STRING,
  value DECIMAL(10,2),
  status STRING,
  create_time TIMESTAMP
);
INSERT INTO qa_test.table_a VALUES (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');
```

> **SQL 消费方式**：前置条件中的 SQL 将由 `dtstack-cli` 包的 `setupPreconditions` 自动执行（DDL 建表 → 数据源导入资产 → 元数据同步）。
> 默认使用 Doris 语法；每张表 SQL 以 `DROP TABLE IF EXISTS` 开头；必须包含 `INSERT INTO` 样例数据。
> SQL 文件路径约定：`tests/e2e/YYYYMM/{suite}/sql/{name}.sql`。

### 9.2 血缘 / 同步 / 回刷类场景额外约束

出现以下关键词时，前置条件还必须包含「链路如何建立」：

- `血缘`、`上下游`、`解析`
- `脚本模式指标`、`向导模式指标`
- `同步`、`调度任务`
- `历史数据`、`升级`、`补数`、`回刷`

此类场景前置条件至少包含：

1. **执行目标**：在哪个引擎 / 集群执行；Doris2.x 和 Doris3.x 复用相同库表名时必须明确集群
2. **建表 / 灌数 SQL**：真实表结构与真实业务数据
3. **建链 SQL / 任务 SQL**：如 `INSERT OVERWRITE`、调度任务 SQL、解析链路所依赖的实际 SQL
4. **指标定义 / 脚本 SQL**：若涉及指标，给出原子 / 派生 / 复合指标来源，或完整脚本模式 SQL
5. **历史回刷 SQL**：若验收点涉及历史兼容，给出历史样例准备方式或回刷 SQL
6. **执行顺序**：若结果依赖操作顺序，必须写明先后关系

**禁止写法：**

- `确保它们与 xxx 建立可追溯血缘关系`
- `准备历史指标样例`
- `同步相关数据后即可验证`

### 9.3 多步表单操作必须完整闭合

当前置条件涉及多步表单（如 Step 1 → Step 2 → Step 3）或多阶段向导时，**必须描述每一步的完整配置直至最终保存**。禁止写到某一步就中断。

**检查方法**：若前置条件中出现「下一步」「Step 2」「第二步」等关键词，则必须能找到对应的后续步骤描述和最终保存/提交操作。

| 禁止写法                                                                 | 正确写法                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

## 12. 可读性格式规范（强制）

### 12.1 表单填写步骤必须换行列举

同一表单/面板内的配置项必须合并为一个步骤，且**必须包含表单中所有的配置项**（不能仅填写必填项），按人类视觉从上到下、从左到右的顺序排列。**在 JSON 输出中，必须使用 `\n` 换行符 + `- 字段: 值` 列表格式，禁止将所有字段用逗号/顿号堆在一行。**

**格式规范**：

- 必填项前加 `*` 标记
- 选项类字段直接给出选中值即可，不要加多余的描述（如「在下拉列表中选择」）
- 非必填字段不填时标注 `无`
- 步骤开头用「点击【入口按钮名】，配置如下：」格式
- 末尾以「点击【保存/确定】按钮」结束

✅ 渲染后效果：

```text
点击【添加规则-完整性校验】，配置如下：
- *规则类型: 字段级
- *字段: info
- *统计函数: key范围校验
- 过滤条件: 无
- *校验方法: 包含
- *校验内容: key1（姓名）和 key2（年龄）
- 强弱规则: 弱规则
- 规则描述: 无
点击【保存】按钮
```

✅ 对应的 JSON `step` 字段写法（**注意 `\n` 换行**）：

```json
{
  "step": "点击【添加规则-完整性校验】，配置如下：\n- *规则类型: 字段级\n- *字段: info\n- *统计函数: key范围校验\n- 过滤条件: 无\n- *校验方法: 包含\n- *校验内容: key1（姓名）和 key2（年龄）\n- 强弱规则: 弱规则\n- 规则描述: 无\n点击【保存】按钮"
}
```

❌ 错误写法 1 — 所有字段堆在一行：

```json
{
  "step": "点击【添加规则】，在表单中填写规则类型为\"字段级\"、字段为\"info\"、统计函数为\"key范围校验\"、校验方法为\"包含\"、校验内容为\"key1（姓名）和 key2（年龄）\"，然后点击【保存】按钮"
}
```

❌ 错误写法 2 — 格式不规范：

```text
- 统计函数: 在下拉列表中选择"key范围校验"
- 字段: 在字段选择框中选择"info"（json类型字段）
- 校验方法: 选择"包含"
- 强弱规则: 保持"强规则"
```

**错误原因**：

1. 将所有字段用逗号连在一行，未使用 `\n` + `- 字段: 值` 列表格式
2. 缺少必填项标记 `*`
3. 字段排列顺序不按页面视觉顺序
4. 选项值前有冗余描述（「在下拉列表中选择」「在字段选择框中选择」）
5. 缺少部分非必填字段（过滤条件、规则描述）

### 12.2 多项内容必须编号换行

前置条件、步骤描述、预期结果中存在多个子项时，必须通过 `1)` `2)` 编号换行，禁止用逗号连接多个子项堆在一行。

2) 二至五层Sheet各自包含「*上一层级的key名」、「*key」、「中文名称」、「value格式」列

❌ 浏览器下载文件，文件命名为 json_format_import_template.xlsx，打开文件后包含5个Sheet（分别对应一层、二层、三层、四层、五层），一层Sheet包含「*key」、「中文名称」、「value格式」列，二至五层Sheet各自包含「*上一层级的key名」、「*key」、「中文名称」、「value格式」列

### 12.5 大数据量场景必须提供数据生成脚本

涉及大量测试数据生成的用例（如千级 key、百级数据行、批量导入等），前置条件中必须提供可执行的数据生成脚本（Python），用于快速生成测试数据文件或 SQL。

```text
✅ 1) 使用 admin 账号登录系统
2) 通过【导入】功能批量导入1000条以上key数据到json格式校验管理列表，导入文件包含1000行一层key数据（key_0001 至 key_1000），每条记录数据源类型为 sparkthrift2.x，数据生成脚本如下:

import openpyxl
wb = openpyxl.Workbook()
ws = wb.active
ws.title = "一层"
ws.append(["key", "中文名称", "value格式"])
for i in range(1, 1001):
    ws.append([f"key_{i:04d}", f"测试key{i}", ""])
wb.save("json_format_import_1000.xlsx")

❌ 1) 使用 admin 账号登录系统
2) 通过【导入】功能批量导入1000条以上key数据到json格式校验管理列表，导入文件包含1000行一层key数据（key1 至 key1000），每条记录数据源类型为 sparkthrift2.x
```

**适用场景**：

- 导入文件包含大量行数据（≥100行）
- SQL 灌数涉及大量记录（≥100条）
- 批量创建/生成测试数据

**脚本要求**：

- 使用 Python 编写，依赖库仅限 `openpyxl`（Excel）或标准库
- 脚本可直接复制执行，无需额外配置
- 生成的文件命名与用例中引用的文件名一致
