## 核心编写规则（必须严格遵守）

### 输出格式
输出必须为纯 JSON，完整 Schema 见 `references/intermediate-format.md`。关键要求：
- `meta.agent_id` = `"writer-[模块简称]"`；`meta.tags` 3-8个领域关键词；`meta.module_key` 填模块 key
- `meta.prd_version`：若 PRD frontmatter 中有 `prd_version` 字段，从中读取（如 `"v6.4.10"`）；archive/xmind 输出目录统一使用年月制（如 `cases/archive/202604/`）
- `meta.version`：有版本号时与 `prd_version` 保持一致（如 `"v6.4.10"`）；无版本号时用日期格式（如 `"202603版本"`）
- 层级：modules[].name（L2 菜单/模块名）→ pages[].name（L3 页面名）→ sub_groups[].name（L4 可选）→ test_cases
- 页面功能单一或用例数 ≤ 5 条时，可跳过 sub_groups，直接在 pages 下放 test_cases

### 层级命名规则（关键！）

modules[].name、pages[].name、sub_groups[].name 必须反映系统中的**实际菜单/页面/功能**，不得使用需求描述、优先级分组或功能概述。

**正确 vs 错误示例：**

| 层级 | ✅ 正确 | ❌ 错误 |
|------|---------|---------|
| modules[].name（L2 菜单名） | `"商品管理"` | `"电商平台"（太宽泛，是功能域不是菜单）` |
| modules[].name（L2 菜单名） | `"订单处理"` | `"商品列表-状态管理"（这是需求描述）` |
| pages[].name（L3 页面名） | `"列表页"` | `"商品管理-状态筛选（逻辑说明）"（这是需求描述）` |
| pages[].name（L3 页面名） | `"新增商品页"` | `"P0 冒烟-主流程"（这是优先级分组）` |
| sub_groups[].name（L4 功能子组） | `"搜索"` `"字段校验"` `"新增商品"` | `"P0 冒烟-主流程"（禁止按优先级分组）` |
| sub_groups[].name（L4 功能子组） | `"商品分类配置"` `"价格配置"` | `"模式切换交互"（太模糊）` |
| sub_groups[].name（L4 功能子组） | `"新增操作"` `"编辑操作"` `"删除操作"` | `"主流程"（禁止使用，这是泛指，不是功能名）` |

> **sub_groups 的存在判断**：当 page 下仅有 1 种功能时，直接在 page 下放 test_cases，不要强行套一个 sub_group。`"主流程"` 通常是"只有一类操作但硬套了分组"的信号，遇到时应直接展平。

**判断口诀**：如果这个名字不能在系统左侧菜单或页面 Tab 中找到对应位置，就不能用作 modules/pages 的名称。

### 前置条件（precondition）

**普通用例**：如无特殊前提，可填"无"。

**当需求涉及数据表操作、数据接入、批量同步、任务调度、对账、规则配置、血缘解析、指标生成、历史升级/补数/回刷等能力时**：**必须**给出具体、可执行的数据准备说明，**与 config.repos 是否为空无关**。PRD 中提到哪些数据源类型，就要在前置条件中给出哪些数据源的建表语句与关键执行 SQL。

前置条件模板：
```
1、环境说明：目标模块已部署，数据源连接正常

2、Doris2.x SQL语句准备:
DROP TABLE IF EXISTS qa_test.table_a;
CREATE TABLE qa_test.table_a (
  id INT,
  name VARCHAR(100),
  value DECIMAL(10,2),
  status VARCHAR(20),
  create_time DATETIME
);
INSERT INTO qa_test.table_a VALUES
  (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00'),
  (2, '测试数据B', 200.00, '停用', '2026-01-02 10:00:00');

3、Hive2.x SQL语句准备:
CREATE TABLE IF NOT EXISTS qa_test.table_a (
  id INT,
  name STRING,
  value DECIMAL(10,2),
  status STRING,
  create_time TIMESTAMP
);
INSERT INTO qa_test.table_a VALUES
  (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');
```

建表语句要求：
- 表名、字段名、字段类型必须与 PRD 中描述的业务场景一致（如 schema 用 `qa_test`，表名和字段名用真实业务名）
- INSERT 数据必须是符合业务语义的真实测试数据，不能用 `test1`、`abc` 等占位符
- 多数据源场景，每种数据源单独给出（语法差异：Doris 用 `VARCHAR`，Hive 用 `STRING` 等）
- 只列 PRD 中实际涉及的数据源类型，不需要穷举所有类型

**当需求属于血缘 / 同步 / 回刷 / 解析类场景时，前置条件还必须额外包含：**
- **执行目标**：在哪个引擎 / 集群执行；如 Doris2.x、Doris3.x 共用相同库表名，必须明确“本步骤在哪个集群执行”
- **建链 SQL / 任务 SQL**：如 `INSERT OVERWRITE`、调度任务 SQL、解析链路依赖 SQL。**禁止**只写“确保建立血缘关系”
- **指标定义 / 脚本 SQL**：如原子/派生/复合指标来源表与字段，或脚本模式完整 SQL
- **历史回刷 SQL**：若验收点包含历史兼容、升级后可用，必须给出回刷 SQL 或历史样例准备方式
- **执行顺序**：若依赖“先建链再同步”“先下线再刷新”，必须写出顺序

**前置条件中禁止出现以下模糊写法：**
- `确保它们与 xxx 建立可追溯血缘关系`
- `保留查询结果的上游表记录`
- `准备历史指标样例`
- `同步相关数据后即可验证`

出现上述语义时，必须改写为可执行语句或明确动作。

### 步骤格式
- **第一步必须是**：进入【完整导航路径】页面
  - ✅ 单层：`进入【商品列表】页面`
  - ✅ 多层：`进入【商品管理 → 商品列表】页面`
  - ❌ 禁止：`进入【商品列表】页面（导航路径：...）`（导航路径应在【】内，不作为括号注释）
- 步骤描述禁止包含「步骤1:」「步骤2:」「Step1:」等编号前缀
- 表单填写必须给出具体测试数据（禁止「填写相关信息」「选择合适选项」等模糊表述）
- 步骤数量必须与预期结果数量完全相等

### 表单多字段步骤合并规则（强制，不可违反）

**同一表单/面板内的所有配置项必须合并为一个步骤节点**，按字段在页面上的实际渲染顺序（参考前端源码 Form 组件定义顺序）逐项列出。在 JSON 中换行使用 `\n` 转义字符。

**✅ 正确（所有字段合并为一个步骤节点）：**
```json
{
  "step": "在【新增商品】表单页面，填写商品信息如下：\n- 商品名称: 2026春季新款运动鞋\n- 商品分类: 运动鞋（下拉选择）\n- SKU编号: SKU-2026-001\n- 价格: 299.00\n- 库存数量: 100\n- 状态: 上架\n点击【保存】按钮",
  "expected": "页面提示"新增成功"，列表页新增一条记录，商品名称显示为"2026春季新款运动鞋""
}
```

**❌ 禁止（每个字段单独一个步骤节点）：**
```json
{ "step": "在「商品分类」下拉框选择「运动鞋」", "expected": "商品分类已选择" },
{ "step": "在「商品名称」输入框输入「2026春季新款运动鞋」", "expected": "名称已填写" },
{ "step": "在「价格」输入框输入「299.00」", "expected": "价格已填写" }
```

**格式细则：**
- 字段顺序：与前端源码 Form 组件中 `<FormItem>` 或 `<Form.Item>` 的定义顺序一致
- 非必填字段：保留在列表中，值后加注 `（非必填）` 或 `（可填写无）`
- 嵌套子配置：在父字段后用缩进行继续列举（如「计算逻辑配置」下的「对比方法」）
- 预期结果：当包含多个验证点时，同样用 `\n- ` 分行

**例外（允许单独步骤）**：
- 字段选择会触发页面联动/刷新，且联动结果本身是用例的核心验证点
- 该字段的可选项列表就是被测内容（如专门验证下拉选项范围的用例）

### 源码验证（当 source-context.md 存在时，不可跳过）

**在编写任何步骤之前**，若 source-context.md 路径非「无」，Read 该文件并完成以下验证：

1. **确认导航路径**：使用文件中「1.1 菜单/路由路径」表中的菜单层级（如 `商品管理 → 商品列表` 而非臆造的 `商品页面`）
2. **确认按钮名称**：使用文件中「1.2 按钮文案」表中的实际按钮文案（如「新增商品」而非「添加商品」）
3. **确认表单字段**：使用文件中「1.3 表单字段」表中的 label 文本（如「商品名称」「商品分类」而非「名称」「类型」）
4. **确认表单流程**：若文件中「1.4 多步骤向导」非空，必须将每个步骤拆分为独立的用例步骤，不得压缩为单步
5. **确认关键交互与前置条件模板**：若文件中存在「1.5 关键交互补充」「2.5 测试编写直接可用结论」「2.6 测试可直接复用的前置条件模板」，必须优先复用其中的菜单规则、禁用规则、建链顺序、脚本 SQL 要素与执行口径

**禁止凭 PRD 描述臆造按钮名和字段名。** 如果 source-context.md 中某项标注为「未找到」，直接按 PRD 原文描述编写用例；**严禁**在 precondition、steps 或 expected 中写入任何 `[待核实]`、`[待确认]`、`[源码未找到]` 等开发背景注释——这类信息属于工作流内部状态，不属于用例内容。

### 范围限制
- 仅编写黑盒功能测试用例（测试人员可在页面手动操作完成）
- 禁止出现：接口/HTTP/SQL注入/XSS/并发/响应时间等词汇
- 按钮名称、字段名称必须与源码和 PRD 原型图一致（源码优先，冲突时标注）
- 异常用例每条只能有一个逆向条件

### 用例设计原则
- 正常用例：可包含多个正向条件组合（如：填写全部必填字段后成功提交）
- 异常用例：**仅允许一个逆向条件**（如：仅「商品名称」为空时无法提交）
- 边界用例：针对边界值的单一验证（如：输入最大长度字符）

#### 正向用例合并规则（强制）

**同一功能区（sub_group 或 page）内，步骤数少于4条的正向用例必须与其他正向用例合并，不得单独成条。**

合并原则：
- 按功能区合并：将同一 sub_group/page 内所有步骤数 < 4 的正向用例，整合为 1 个综合正向用例
- 综合用例的步骤顺序按用户操作流自然衔接（如：进入页面 → 验证默认状态 → 执行主操作 → 验证结果）
- 综合用例前置条件取各子用例前置条件的并集（最全的数据准备）
- **禁止**为了达到"每条用例步骤少"而将一个自然流拆为多条独立用例

**反例**（禁止）：
```
✗ 用例A（1步）：验证商品列表页默认加载成功
✗ 用例B（1步）：验证商品列表页列字段完整显示
✗ 用例C（1步）：验证商品列表页状态筛选项默认展示
✗ 用例D（3步）：验证商品列表页分页功能正常
```

**正例**（正确）：
```
✓ 用例（4步）：验证商品列表页默认加载、列字段完整显示、状态筛选及分页功能
  步骤1：进入【商品管理 → 商品列表】页面 → 验证加载状态、按钮、列字段（商品名称/分类/价格/状态）
  步骤2：在「状态」筛选项选择「上架」→ 列表仅显示上架商品
  步骤3：点击下一页 → 分页切换正常
  步骤4：切换每页条数为50 → 列表重新加载，每页显示50条
```

异常用例不适用此规则——每条异常用例仍只测试一个逆向条件，即使步骤数只有1条也不与其他异常用例合并（合并会导致一个用例含多个逆向条件，违反设计原则）。

### 优先级标准

**P0（冒烟，约占总用例 25%）**：
覆盖所有核心功能路径的最小可执行集，保证跑完 P0 即可判断主流程是否可用。
- 每个页面/弹窗的正常打开与加载
- 每类核心操作的正向主流程（新增成功、编辑成功、删除成功、搜索成功、导入成功、导出成功）
- 核心联动逻辑的正向验证（如：字段A填写后字段B正常显示）
- 关键展示类验证（如：列字段完整性、Tab切换后列表刷新、树形默认展开状态）

**P1（核心功能，约占总用例 50%）**：
- 必填字段为空的校验（异常用例）
- 字段长度/格式校验（异常用例）
- 业务规则分支（如：含子层级时的级联删除、第5层无新增子层级按钮）
- 联动逻辑的反向验证（如：字段A清空后字段B隐藏）
- 搜索/筛选的各条件分支

**P2（扩展场景，约占总用例 25%）**：
- 边界值（最大/最小字符数）
- 取消操作、弹窗关闭不保存
- 搜索无结果等边界状态
- 非必填字段不填时正常提交
- 导出文件命名等细节验证

## 自评审清单（输出前逐项检查）

- 标题以「验证」开头；第一步为「进入【完整导航路径】页面」
- 无「步骤N:」编号前缀；步骤数 == 预期数；无模糊词（尝试/如/比如/等）
- 所有填写步骤有具体测试数据；预期结果具体描述系统行为
- 按钮/页面名与 PRD 一致；异常用例只有一个逆向条件
- 无性能/安全/接口内容；未超出本次需求范围
- precondition / steps / expected 中**不含**任何 `[待核实]`、`[待确认]`、`[源码未找到]` 等开发背景注释
- modules[].name 为系统中实际菜单名；pages[].name 为页面类型名（列表页/新增页/编辑页/详情页）
- sub_groups[].name 为功能子组名（搜索/字段校验/导出），不得使用优先级标签（P0/P1/P2）
- 多字段步骤已用 `\n- ` 换行，不是一句话压缩
- 涉及数据表操作/数据接入/批量同步/调度/规则配置/血缘解析/历史回刷的用例，前置条件包含具体建表、INSERT、建链 SQL / 任务 SQL、指标 SQL / 回刷 SQL 与执行口径

## 输出要求

将 JSON 写入文件：[临时文件路径，如 cases/prds/202604/temp/list.json]

然后输出简短摘要：
- 共编写 X 条用例（P0: N / P1: N / P2: N）
- 覆盖的功能点：[列表]
- 自评审结果：全部通过 / [问题描述]
```
