# Step req-elicit：需求澄清

> 前置条件: `last_completed_step` == `"parse-input"`
> 快速模式: **简化执行**（最多 3 个问题，仅 Tier 1 维度，1 轮 Q&A）

本步骤在进入流水线执行之前，先评估需求的可测试性，识别关键缺失信息，通过 3-7 个有针对性的问题补全，最终将澄清结果追加到原始 PRD 中。

**执行前必须阅读 `references/elicitation-dimensions.md`（维度定义、评分规则、问题模板）。**

---

## Phase 1: 可测试性评估（自动，无需等待用户）

读取工作目录下所有原始 PRD 文件内容，按 `elicitation-dimensions.md` 中定义的 10 个维度评分：

1. 对每个维度，根据评分规则计算 0-100% 的得分
2. 可选维度（`time_limits`、`tech_constraints`）：检测 PRD 中是否含对应关键词，有则启用，否则标记 `N/A`
3. 计算加权平均分（字段定义 3x、页面清单 2x、验收标准 2x，其余 1x）

**根据评分决定下一步行为：**

| 评分 | 行为 |
|------|------|
| 普通模式 >= 90% | 跳过澄清，直接完成此步骤（输出跳过原因） |
| 快速模式 >= 70% | 跳过澄清，直接完成此步骤 |
| 普通模式 < 40% | 进入 Phase 2，但先输出「目标/已知/缺失/建议」摘要 |
| 其他 | 进入 Phase 2 |

**跳过时的输出：**

```
需求可测试性评分 {score}%，信息充分，跳过需求澄清步骤。

维度得分概览：字段定义 {fd}% | 页面清单 {pi}% | 验收标准 {io}% | 其他均 >= 80%
```

## Phase 2: 源码预扫描（当 config.repos 非空时，轻量）

**触发条件**: config.json 中 `repos` 字段为非空对象，且 `.repos/` 目录下存在已 checkout 的仓库（即使分支可能不是最新的）。

**目的**: 减少需要用户回答的问题数量（源码能回答的，不问用户）。

执行以下轻量扫描（仅用于减少问题，结果标记为 advisory，非权威）：

```bash
# 1. 推断目标分支
cat "<branchMapping-from-.claude/config.json>"  # 结合 PRD 中的 dev_version 推断

# 2. 查找 DTO/VO 类（按 PRD 中的功能名/模块名关键词）
grep -r "class.*{keyword}.*DTO\|class.*{keyword}.*VO\|class.*{keyword}.*Param" \
  .repos/{backend_repo}/src --include="*.java" -l | head -5

# 3. 提取字段和注解
grep -r "@NotNull\|@NotBlank\|@Length\|@Size\|@Min\|@Max\|@Pattern" \
  .repos/{backend_repo}/src/{found_dto_path} | head -30

# 4. 查找枚举定义
grep -r "enum.*{keyword}" .repos/{backend_repo}/src --include="*.java" -l | head -5

# 5. 前端字段标签（如 frontend 仓库存在）
grep -r "label.*[\u4e00-\u9fa5]\|FormItem\|Form.Item" \
  .repos/{frontend_repo}/src -r --include="*.tsx" -l | head -5
```

将找到的信息整理为「已自动推断」列表，供 Phase 3 使用。

**config.repos 为空时（无源码仓库配置）**: 跳过此 Phase，所有字段级问题均需询问用户。

## Phase 3: 问题生成与展示

### 3.1 整理已知信息

收集以下来源的已知信息：
- Phase 1 评估结果（哪些维度得分充足）
- Phase 2 源码预扫描结果（哪些字段信息可自动推断）
- PRD frontmatter（已填写的 `dev_version`、`product` 等字段）

### 3.2 生成问题

**过滤规则**:
- 得分 >= 80% 的维度：不提问
- 源码已明确推断的信息：归入「已自动推断」区，不提问
- 剩余不足的维度：按 `elicitation-dimensions.md` 权重排序，生成问题

**问题数量控制**:
- 普通模式: 3-7 个（不超过 7 个，合并同维度问题为一组）
- 快速模式: 1-3 个（仅 Tier 1：`field_definition`、`page_inventory`、`io_criteria`）
- 若所有高权重维度都充足，可以提出 0 个问题（但评分应已触发跳过）

### 3.3 展示格式：逐页推进

**核心原则：每次只展示 1 个问题，等用户回应后再推进下一条。不一次性发出全部问题。**

#### 3.3.1 开场总览（仅展示一次，不含具体问题）

```
需求可测试性评分：{score}%（字段定义 {fd}%↓ | 页面清单 {pi}% | 验收标准 {io}%↓）

需补充 {N} 项信息（{inferred_count} 项已自动推断待确认，{question_count} 个问题待回答）。
逐条提问中，每次一个，你随时可以：
  全部推荐 — 接受所有 AI 推荐选项，一步完成
  直接开始 — 跳过全部澄清，继续流程
```

（展示完总览后，立即推进到第 1 页，不等待用户回复）

#### 3.3.2 单页格式（推断确认页）

```
┌─ 第 {current}/{total} 项 · 自动推断确认 ─────────────────────┐

【推断】{推断内容的一句话描述}
  {推断依据，如：检测到 dev_version「6.4.10」→ 计划切换至 `release/6.4.10`}

  A  ✓ 确认正确                                   ← 推荐
  B  {否定分支，如「分支有误，我来指定」}
  C  {其他否定分支（若有）}

└──────────────────────────────────────────────────────────────┘
```

#### 3.3.3 单页格式（问题页）

```
┌─ 第 {current}/{total} 项 · {维度名} {星级} ─────────────────┐

{question_text}

  A  {option_a}                                   ← 推荐（{inference_basis}）
  B  {option_b}
  C  {option_c（若有）}
  {倒数第二} 跳过，AI 自动推断
  {末位}  自定义输入...

└──────────────────────────────────────────────────────────────┘
```

**选项生成规则**：

| 选项位置 | 内容 |
|---------|------|
| A（第一项） | AI 最优推断，标注 `← 推荐` 和来源（源码推断/PRD原文/常见模式） |
| 中间项（B/C/...） | 次优或常见替代值 |
| 倒数第二项 | 「跳过，AI 自动推断」（始终存在） |
| 末位 | 「自定义输入...」（始终存在） |

**星级标注**：`★★★` 字段定义 | `★★` 页面清单/验收标准 | `★` 其他维度

**顺序约定**：推断确认项优先（按数量从少到多），再按维度权重排列问题（★★★ 先）。

**注意**:
- 如果没有源码推断结果，跳过推断确认项，直接从 Q1 开始
- 模糊需求（< 40%）在第 1 页展示摘要，然后从第 2 页起逐条推进（见 Phase 4）

## Phase 4: 模糊需求处理

当评分 < 40% 时，在展示问题前先输出结构化摘要：

```
## 需求理解摘要

**根据您提供的内容，我的理解是：**
{一段话描述理解的功能目标}

### 已知条件
- {从 PRD 确认的事实，逐条}

### 关键缺失信息（按影响程度排序）
| 优先级 | 缺失项 | 对用例质量的影响 |
|--------|--------|----------------|
| 高 | {item} | {impact} |
| 中 | {item} | {impact} |

### 建议
1. {最重要的补充建议}
2. {次要建议}

---

```

（紧接着展示 Phase 3 的问题）

---

## Phase 5: 答案收集与逐页推进

### 5.1 单页响应处理

收到用户对当前页的回复后，立即处理并推进到下一页：

```
✓ 已记录：{所选选项的实际内容}

{下一页内容...}
```

**字母选项解析规则：**

| 用户输入 | 处理方式 |
|---------|---------|
| `A` / `B` / `C`（单字母） | 选择当前页对应选项，标记 `confirmed` |
| 末位字母（自定义） | 展示「请直接输入内容：」，等待输入后记录 `user_custom`，再推进下一页 |
| `B`（推断确认页的纠正选项） | 展示「请提供正确的 {推断项}：」，等待输入后记录 `user_corrected`，再推进下一页 |
| 直接输入文字（未选字母） | 视为自定义输入，记录 `user_custom`，推进下一页 |

**快捷指令（任意时刻均可输入）：**

| 指令 | 行为 |
|------|------|
| `全部推荐` | 当前及剩余所有页均选 A 选项，跳过逐页交互，直接进入 Phase 6 |
| `直接开始` / `跳过全部` | 当前及剩余所有页标记 `bypassed`，展示跳过警告后进入 Phase 6 |
| `跳过` / `S` | 仅跳过当前页，当前项标记 `inferred`（使用 A 选项内容），推进下一页 |

### 5.2 全部完成后的确认页

当所有页均回答完毕，展示汇总确认：

```
✓ 澄清完成（{N}/{N}）

已记录信息摘要：
  • 目标分支：{branch}（{来源}）
  • 字段定义：{字段数} 个字段已补充
  • 验收标准：{scenario_count} 个场景已确认
  • 其他：{other_count} 项

可测试性评分：{before}% → {after}%（+{delta}%）

继续进入下一步...
```

（无需用户确认，直接进入 Phase 6）

### 5.3 跟进追问（若用户回答暴露新缺口）

若某页回答中暴露了新的关键缺口，在下一页位置插入追问（不打破分页节奏）：

```
┌─ 第 {current+1}/{total+1} 项 · 跟进 ────────────────────────┐

{追问内容，基于上一条回答}

  A  {option_a}                                   ← 推荐
  B  ...
  跳过  自定义输入...

└──────────────────────────────────────────────────────────────┘
```

全流程最多追加 2 个跟进问题。

### 5.4 跳过警告

当用户选择「直接开始」时展示：

```
⚠️ 已跳过需求澄清。当前可测试性评分 {score}%，以下维度信息不足可能导致：
- 字段定义不完整 → 表单测试用例无法覆盖所有输入场景
- 验收标准缺失 → 异常场景测试用例可能不准确

将继续生成测试用例，建议后续人工核查相关用例。
```

## Phase 6: 输出澄清结果

将澄清结果追加到原始 PRD 文件末尾（**不修改 PRD 原有内容，仅追加新章节**）：

```markdown

---

## 需求澄清结果

> 以下内容基于用户回答和源码分析整理。

<!-- elicitation-status: completed | round: {N} | questions: {asked}/{answered} | auto-inferred: {count} -->

### 目标与背景

- **业务目标**: {用户确认/推断的目标描述}
- **目标用户**: {标品/定制版/等}
- **目标分支**: {branch_name}（来源: {PRD字段/用户确认/AI推断}）

### 使用场景与前置条件

- **数据源类型**: {${datasource_type} / 无数据库依赖}
- **测试数据准备**: {具体说明，或「无特殊要求」}
- **其他前置条件**: {如需先维护商品分类、仓库配置、数据源连接等}

### 字段定义补充

| 字段名 | DTO字段名 | 类型 | 是否必填 | 校验规则 | 枚举值 | 来源 |
|--------|----------|------|---------|---------|--------|------|
| {字段名} | {dtoField} | {类型} | {是/否} | {规则} | {值列表} | {PRD原文/源码推断/用户确认} |

> 注：「来源」列说明信息可信度：「用户确认」> 「源码推断」> 「AI推断」

### 验收标准

| 场景类型 | 场景描述 | 预期结果 | 来源 |
|---------|---------|---------|------|
| 正常 | {操作描述} | {具体预期} | {用户确认/推断} |
| 异常 | {错误场景} | {错误提示/行为} | {用户确认/推断} |
| 边界 | {边界条件} | {预期行为} | {用户确认/推断} |

### 风险与边界

- {识别到的风险或边界条件，逐条列出}

### 可测试性评分

- **澄清前**: {score_before}%
- **澄清后**: {score_after}%（提升 {delta}%）
- **仍待确认**: {remaining_gaps，或「无」}
```

**更新 PRD frontmatter**：将 `status` 从 `未开始` 更新为 `已澄清`。

---

## 错误处理

- **PRD 文件不存在或无法读取**: 提示用户检查文件路径，不继续
- **多个 PRD 文件**: 合并分析所有 PRD，生成统一的澄清问题（而不是分别澄清）
- **源码预扫描失败（grep 报错/仓库不存在）**: 跳过 Phase 2，不提示用户，继续生成问题
- **PRD 已有 `status: 已澄清` 或 `status: 已增强`**: 跳过澄清步骤，说明「需求已澄清，直接继续」
