## 步骤格式正反例

| 错误写法 | 正确写法 |
|----------|---------|
| 步骤1: 进入列表页 | 进入【商品管理 → 商品列表】页面 |
| 尝试提交表单 | 点击【保存】按钮 |
| 输入像"测试"这样的内容 | 在「商品名称」输入框输入"2026春季新款运动鞋" |
| 选择如"上架"等状态 | 在「状态」下拉框选择"上架" |
| 填写相关信息 | 在「商品名称」输入"2026春季新款运动鞋"，在「商品分类」选择"运动鞋" |

## 表单多字段合并规则（强制）

同一表单/面板内的多个配置项 **必须合并为一个步骤节点**，按字段在页面上的实际顺序（参考前端源码）逐项列出。

**✅ 正确写法（一个步骤节点，结构化列举所有字段）：**

```json
{
  "step": "在【新增商品】表单页面，填写商品信息如下：\n- 商品名称: 2026春季新款运动鞋\n- 商品分类: 运动鞋（下拉选择）\n- SKU编号: SKU-2026-001\n- 价格: 299.00\n- 库存数量: 100\n- 状态: 上架\n点击【保存】按钮",
  "expected": "页面提示"新增成功"，列表页新增一条记录，商品名称显示为"2026春季新款运动鞋""
}
```

**❌ 禁止写法（每个字段拆成独立步骤节点）：**

```json
{ "step": "在「商品分类」下拉框选择「运动鞋」", "expected": "..." },
{ "step": "在「商品名称」输入框输入「2026春季新款运动鞋」", "expected": "..." },
{ "step": "在「价格」输入框输入「299.00」", "expected": "..." }
```

**例外情况（允许单独步骤）：**
- 字段选择触发页面联动，且联动结果是本用例的核心验证点
- 该字段本身的选项列表就是被测内容（如验证「状态」下拉框选项仅含「上架」「下架」「待审核」）

## 预期结果正反例

| 错误写法 | 正确写法 |
|----------|---------|
| 操作成功 | 页面提示"新增成功"，列表页新增一条记录 |
| 显示正确 | 列表第一行显示：商品名称"2026春季新款运动鞋"，状态"上架"，价格"299.00" |
| 提交失败 | 「商品名称」下方显示红色提示"请输入商品名称" |

## 数据禁止词

步骤和预期结果中禁止出现以下模糊词汇：
「尝试」「像」「如」「比如」「等」「相关」「合适」「某个」「一些」「适当」

## Tags 推断规则

在 `meta.tags` 中填入 3-8 个领域关键词：
- **纳入**：产品/功能域名词（如 商品管理、订单处理、用户中心）
- **纳入**：业务实体名词（如 商品、订单、用户、SKU、分类）
- **纳入**：客户/项目标识（如项目缩写或产品线名称）
- **排除**：页面级通用词（列表页、新增、编辑、删除、详情、查询、搜索）

`meta.module_key` 填入当前模块的英文 key（如 `product-mgmt`、`order-center`）。

## PRD 矛盾与不确定性处理

- 前后矛盾（如字段描述与原型图不一致）→ 在 `precondition` 末尾标注 `[待核实：PRD 中 xxx 与 xxx 描述不一致]`
- 信息不足 → 跳过该功能点并在输出摘要中注明
- 禁止自行推测 PRD 未提及的功能行为

## 依赖用例处理

- 依赖另一条用例的前置状态 → 在 `precondition` 中明确引用：如 `已成功新增一条商品记录（商品名称：2026春季新款运动鞋）`
- 不要假设其他 Writer 的用例已存在，每条用例的前置条件必须自包含

## 跨模块联动用例正反例

**场景**：PRD 描述了「规则配置」功能，支持新增规则、执行规则，执行后在「实例管理」页面查看运行记录。

**反例（只验证了新建，遗漏了执行和联动）：**

```
用例: 【P0】验证新增规则成功提交
  步骤1: 进入【规则配置 → 规则列表】页面
  步骤2: 点击【新增】→ 填写表单 → 点击【保存】
  预期: 提示"新增成功"，列表新增一条记录
  （结束 — 只有新建，没有执行和验证）
```

**正例（一条用例串联 新建→执行→验证 完整流程）：**

```
用例: 【P0】验证新增规则并执行成功后生成运行实例
  步骤1: 进入【规则配置 → 规则列表】页面
  预期: 页面正常加载

  步骤2: 点击【新增】按钮，填写规则信息如下：
    - 规则名称: 数据质量校验规则-001
    - 规则类型: 字段校验
    - 关联数据源: Doris2.x-测试集群
    - 校验表: qa_test.table_a
    点击【保存】按钮
  预期: 提示"新增成功"，规则列表新增一条记录，状态为"待执行"

  步骤3: 在规则列表中，点击"数据质量校验规则-001"行的【执行】按钮
  预期: 提示"已提交执行"，该规则状态变为"运行中"

  步骤4: 进入【实例管理】页面，在搜索框输入"数据质量校验规则-001"
  预期: 实例列表显示一条记录：
    - 规则名称：数据质量校验规则-001
    - 状态：运行中（或已完成）
    - 触发时间：显示当前时间

  步骤5: 等待执行完成后，刷新实例列表页面
  预期: 实例状态变为"成功"，运行时长显示具体耗时

  步骤6: 点击该实例行的【详情】按钮
  预期: 实例详情页正常加载，显示执行日志、输入输出参数、校验结果明细
```

**判断口诀**：如果 PRD 中的某条记录有「执行/运行/触发」入口，但你的用例只写到了「创建成功，列表有这条记录」就结束了，那就是遗漏了联动场景。一条用例应当串联从新建到执行到验证结果的完整流程。

## 源码分析（详细版）

有源码时必须执行以下 5 项：

1. **接口与字段名验证**：Grep 搜索 Controller 中的接口路径和方法名，确认 PRD 中的按钮/操作对应实际接口
2. **DTO/VO 字段名提取**：找到请求/响应 DTO 类，提取字段名及其注解（`@NotNull`、`@Length`、`@Pattern` 等），确保步骤中使用的字段名与实际一致
3. **校验规则提取**：从 DTO 校验注解和 Service 层校验逻辑中提取异常边界条件（如最大长度、必填规则、枚举范围）
4. **业务逻辑分支**：阅读 Service 层核心方法，识别 if/else 分支和异常抛出点
5. **联动关系识别**：检查 Service 层的跨模块调用（如新增商品后触发库存初始化）

## 源码额外要求（当 config.repos 非空时）

当 source_context 不为空时必须执行：
- 先确认 `.repos/` 中对应仓库已切到 source_context 指定分支
- **前置条件必须包含数据准备语句**：PRD 中提到哪些数据源类型（`${datasource_type}` 等），就必须给出对应数据源的建表和插入测试数据语句。格式参见 writer-subagent.md 的「前置条件模板」
- 复杂表单步骤**必须**使用多行结构化块（JSON 中用 `\n- ` 换行），禁止压缩为一句话
- 步骤中涉及的数据源名称、数据库名、表名必须与前置条件中的建表语句一致
- 关联字段、主键等配置步骤也要用结构化块逐项列出，示例：
  ```json
  {
    "step": "在关联商品配置区域点击【添加关联商品】按钮，配置如下：\n- 关联商品名称：选择「经典白色帆布鞋」\n- 关联类型：选择「搭配推荐」\n- 排序权重：填写「1」",
    "expected": "新增一行关联商品配置，三个字段均正常选择，点击【保存】后关联关系建立成功"
  }
  ```
