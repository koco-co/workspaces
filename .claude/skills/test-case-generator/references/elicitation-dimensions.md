# 需求澄清维度定义（Elicitation Dimensions）

> 供 `step-req-elicit` 和 `prd-enhancer Step 0.3` 使用。
> 定义可测试性评分的 10 个维度、评分规则、自动推断来源和问题模板。

---

## 评分公式

**加权平均分 = Σ(维度得分 × 权重) / Σ(已启用维度权重)**

- 可选维度（`time_limits`、`tech_constraints`）：仅在需求中检测到对应关键词时启用，否则标记 `N/A`，不参与计算

**评分阈值与行为**

| 分段 | 行为 |
|------|------|
| >= 90% | 自动跳过澄清（快速模式 >= 70%），直接进入下一步 |
| 70-89% | 提出 1-3 个补充问题 |
| 40-69% | 提出 3-7 个问题（正常模式），1-3 个（快速模式） |
| < 40% | 先输出「目标/已知/缺失/建议」结构化摘要，再提问 |

---

## 10 个维度

### D1: 目标/业务价值（goal）

**权重**: 1x | **必须**: 是

**评分规则**:
- 100%: 有明确的「需求背景」章节，说明用户痛点和功能目标
- 60%: 有功能描述但缺业务背景（如只说"新增质量规则类型"，未说明为什么)
- 30%: 仅有功能名称，无任何背景或目标描述
- 0%: 完全无法判断这个需求要解决什么问题

**自动推断**: 从 PRD 标题、`## 需求背景` 章节、蓝湖提取文本的开头段落推断

**问题模板**:
- Q-G1: 这个功能主要帮助用户解决了什么问题？（一句话）
- Q-G2: 目标用户是内部运维人员、数据开发人员，还是业务人员？

---

### D2: 目标用户/分支（target_user）

**权重**: 1x | **必须**: 是

**评分规则**:
- 100%: `dev_version` 字段已填，且能通过 `.claude/config.json` 的 `repoBranchMapping` 字段解析到具体分支
- 70%: `dev_version` 已填但无法匹配映射（需用户确认）
- 30%: 未填 `dev_version` 但从 PRD 内容（如「岚图定制化」「袋鼠云标品」等关键词）可推断
- 0%: 无任何客户/版本信息

**分支推断规则**（从 PRD 内容关键词匹配）:
- 包含「岚图」「ltqc」→ 推断 `release_6.3.x_ltqc` 分支，`ltqc-custom` profile
- 包含「袋鼠云」「标品」「release_6.」→ 推断 `dtstack-default` profile
- xyzh/信永中和 → 无源码，无需分支

**自动推断来源**: PRD frontmatter `dev_version` + `.claude/config.json` 的 `repoBranchMapping` 字段关键词匹配

**问题模板**:
- Q-TU1: 这个需求是针对哪个客户或产品版本？（标品/岚图定制/其他客户）
- Q-TU2: 对应的开发版本是什么？（如「6.3岚图定制化分支」「v6.4.10」）

---

### D3: 使用场景/前置条件（usage_scenario）

**权重**: 1x | **必须**: 是

**评分规则**:
- 100%: 明确说明了数据源类型（Doris/Hive/SparkThrift/etc）、测试表结构或数据准备要求
- 60%: 有部分前置条件但数据源类型模糊（如只说「数据质量任务」）
- 30%: 有功能场景描述但无任何测试数据或环境要求
- 0%: 无使用场景描述

**自动推断**: 从 PRD 中的「前置条件」「测试环境」「依赖」章节提取；从功能关键词推断（如含「数据质量」→ 需要 Doris/Hive 数据源）

**问题模板**:
- Q-US1: 测试这个功能需要准备哪种数据源？（Doris/Hive/SparkThrift/其他）
- Q-US2: 测试数据有特殊结构要求吗？（如 json 字段、特殊字符、大数据量等）
- Q-US3: 是否有依赖的前置功能或配置？（如需要先创建规则集、数据源连接等）

---

### D4: 页面清单（page_inventory）

**权重**: 2x | **必须**: 是

**评分规则**:
- 100%: 所有提到的页面都有对应的 `####` 级标题和设计说明（列表页/新增页/编辑页/详情页等）
- 70%: 大部分页面有设计说明，1-2 个页面仅被提及但无设计
- 40%: 有原型图但无文字说明，或只有笼统的功能描述
- 0%: 完全无页面拆分，只有需求描述

**自动推断**: 统计 PRD body 中 `####` 级标题数量，识别「列表页」「新增」「编辑」「详情」「弹窗」等关键词出现次数

**问题模板**:
- Q-PI1: 这个功能包含哪些页面？（如：列表页、新增表单页、详情弹窗）
- Q-PI2: 「{page_name}」页面有独立的入口（菜单），还是从其他页面弹出/跳转？

---

### D5: 字段定义表（field_definition）

**权重**: 3x（最高） | **必须**: 是

**评分规则**:
- 100%: 每个表单/列表都有完整字段定义表，包含：字段名 + 字段类型 + 是否必填 + 校验规则 + 枚举值
- 80%: 有字段定义表但部分字段缺少枚举值或校验规则
- 50%: 有字段列表但无结构化定义（如只在正文中提到字段名）
- 20%: 有截图但无文字描述字段
- 0%: 完全无字段信息

**自动推断来源（DTStack）**:
- DTO/VO 类中的 `@NotNull` `@NotBlank` `@Length` `@Size` `@Min` `@Max` `@Pattern` 注解 → 必填和校验规则
- enum 类定义 → 枚举值
- 前端 `FormItem` 的 `label` 属性 → 字段中文名
- `placeholder` 文本 → 字段提示和约束说明

**问题模板**:
- Q-FD1: 「{page_name}」页面表单中有哪些字段？请列出：字段名、是否必填、类型（文本/下拉/日期/等）
- Q-FD2: 「{field_name}」下拉字段有哪些可选值？（完整枚举列表）
- Q-FD3: 「{field_name}」字段有字符长度限制或格式要求吗？
- Q-FD4（源码感知）: 从源码发现 {dto_class} 包含以下字段：{field_list}。PRD 未提及的字段 {missing_fields} 是否需要测试覆盖？

---

### D6: 输入输出/验收标准（io_criteria）

**权重**: 2x | **必须**: 是

**评分规则**:
- 100%: 每个核心操作都有明确的成功/失败判定条件（如「提交成功后列表新增一条记录，状态为待处理」）
- 70%: 有正向描述但缺少异常/失败场景的预期
- 40%: 只有功能说明，无具体的验收条件
- 0%: 完全无验收标准

**自动推断**: 从「验收标准」「预期结果」「成功条件」章节提取；从接口返回值推断（如 HTTP 200 + 特定响应结构）

**问题模板**:
- Q-IO1: 「{operation}」操作成功后，用户会看到什么？（如：提示语、列表变化、状态变更）
- Q-IO2: 「{operation}」操作失败时（如必填项为空、数据重复），系统会如何提示用户？
- Q-IO3: 这个功能的「核心验收标准」是什么？一句话描述判断功能是否正常工作的条件

---

### D7: 业务规则/交互规则（business_rules）

**权重**: 1x | **必须**: 是

**评分规则**:
- 100%: 明确说明了字段联动规则、状态流转、权限限制、数据唯一性约束等
- 60%: 有部分规则说明但不完整（如提到了唯一性但未说明校验时机）
- 30%: 有截图能推断一些规则但无文字说明
- 0%: 无任何业务规则说明

**自动推断**: 从 Service 层的 if/switch 逻辑、`@Unique` 注解、前端联动代码推断

**问题模板**:
- Q-BR1: 表单字段之间有联动关系吗？（如：选了 A 后，B 字段的选项会变化）
- Q-BR2: 这个功能有数据唯一性约束吗？（如：规则名称不能重复）
- Q-BR3: 有审批流程或状态流转吗？（如：草稿→待审批→已发布）

---

### D8: 性能限制（time_limits）

**权重**: 1x | **必须**: 否（仅当检测到性能关键词时启用）

**启用条件**: PRD 中包含以下关键词之一：「性能」「超时」「并发」「响应时间」「毫秒」「QPS」「TPS」「大数据量」「百万」「千万」

**评分规则**:
- 100%: 有具体的性能指标（如「列表加载 < 3s」「支持 1000 并发」）
- 50%: 提到了性能要求但无量化指标
- 0%: 提到性能场景但无任何要求

**问题模板**:
- Q-TL1: 对于「{feature}」，有响应时间要求吗？（如：页面加载 < 3 秒）
- Q-TL2: 需要测试大数据量场景吗？（如：表格有 10 万条记录时的性能）

---

### D9: 技术约束（tech_constraints）

**权重**: 1x | **必须**: 否（仅当检测到技术约束关键词时启用）

**启用条件**: PRD 中包含以下关键词之一：「API」「接口」「兼容」「升级」「迁移」「版本」「限制」「不支持」「只支持」

**评分规则**:
- 100%: 明确说明了技术限制（如「仅支持 Doris 2.x」「不兼容旧版配置」）
- 50%: 提到了技术相关信息但约束不清晰
- 0%: 未说明

**问题模板**:
- Q-TC1: 「{feature}」对数据库版本有要求吗？（如：仅支持 Doris 2.x+）
- Q-TC2: 这个变更是否影响现有功能的行为？（向后兼容性问题）

---

### D10: 风险与边界（risks_boundaries）

**权重**: 1x | **必须**: 是

**评分规则**:
- 100%: 明确说明了边界条件（如字段最大长度、最大列表条数、特殊字符处理）和潜在影响（如历史数据兼容性）
- 60%: 有部分边界说明但不完整
- 30%: 有截图中可见一些边界信息（如 placeholder 文本）
- 0%: 无任何边界或风险说明

**自动推断（DTStack）**: 从 `@Length` `@Size` `@Min` `@Max` 注解推断边界值；从 git diff 推断历史功能变更范围

**问题模板**:
- Q-RB1: 「{field_name}」字段的最大值/最大长度是多少？
- Q-RB2: 这个功能是否影响已有的「{existing_feature}」？有迁移或兼容性问题吗？
- Q-RB3: 有什么操作是不允许的？（如：不允许删除已引用的配置，不允许修改已发布的记录）

---

## 模糊需求处理（评分 < 40%）

当总评分 < 40% 时，跳过提问，先输出结构化摘要：

```markdown
## 需求理解摘要

**我对这个需求的理解是：**
[用一段话描述从 PRD 中理解的功能目标]

### 已知条件
- [从 PRD 中确认的事实，逐条列出]

### 关键缺失信息
| 优先级 | 缺失项 | 影响 |
|--------|--------|------|
| 高 | 字段定义表（完全缺失） | 无法设计表单相关测试用例 |
| 高 | 验收标准 | 无法判断正常/异常用例的预期结果 |
| 中 | 目标用户/客户版本 | 无法确定参考哪个源码分支 |

### 建议
1. [最重要的补充建议]
2. [次要建议]

请补充上述信息后，我将继续澄清并正式输出需求文档。
也可以直接回答，我会将答案整合到文档中。
```

---

## 问题生成优先级排序

当多个维度得分不足时，按以下顺序优先提问（权重高的优先）：

1. `field_definition`（3x）— 字段定义
2. `page_inventory`（2x）— 页面清单
3. `io_criteria`（2x）— 验收标准
4. `goal`（1x）— 目标背景
5. `target_user`（1x）— 目标用户
6. `usage_scenario`（1x）— 使用场景
7. `business_rules`（1x）— 业务规则
8. `risks_boundaries`（1x）— 风险边界
9. `time_limits`（1x，可选）
10. `tech_constraints`（1x，可选）

**合并规则**: 同一维度的多个问题合并为一组，避免碎片化提问。

---

## Smart Skip 规则（源码可自动推断时跳过）

| 维度 | 自动推断条件 | 跳过条件 |
|------|------------|---------|
| `target_user` | `dev_version` 字段已填且能匹配 repo-branch-mapping.yaml | 跳过，显示「已推断」 |
| `field_definition` (必填列) | DTO 中找到 `@NotNull` / `@NotBlank` 注解 | 对应字段跳过必填问题 |
| `field_definition` (枚举值) | enum 类找到完整枚举 | 对应字段跳过枚举问题 |
| `field_definition` (长度) | `@Length` / `@Size` / `@Max` 注解找到 | 对应字段跳过长度问题 |
| `business_rules` (唯一性) | `@Unique` 注解或 Service 层 duplicate check 找到 | 跳过唯一性问题 |

自动推断的信息展示在「已自动推断（请确认）」区，用户可纠正。
