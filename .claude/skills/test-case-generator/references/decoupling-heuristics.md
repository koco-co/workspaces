# 需求解耦判断规则

本文件定义了将 PRD 中的功能模块拆分为并行 Subagent 任务的判断规则。

---

## 可并行条件（满足任一即可并行）

| 条件 | 说明 | 示例 |
|------|------|------|
| 不同 UI 页面 | 在不同的独立页面上操作 | 质量问题台账 → 列表页 vs 质量问题台账 → 新增页 vs 质量问题台账 → 详情页 |
| PRD 不同编号章节 | 在 PRD 中属于不同的 4.x 章节 | 4.1 质量问题台账 vs 4.2 安全问题台账 |
| CRUD 不同操作 | 增/查/改/删各自独立 | 新增表单 vs 搜索条件 vs 编辑弹窗 vs 删除确认 |
| 不同数据对象 | 操作的是不同的业务实体 | 台账记录 vs 附件管理 |

## 需串行条件（满足任一则串行）

| 条件 | 说明 | 示例 |
|------|------|------|
| 前置数据依赖 | 一个模块的前置条件是另一个模块的输出结果 | 详情查看依赖新增成功的数据 |
| 共享工作流状态 | 一个模块的操作会影响另一个模块的可见性/可操作性 | 状态为「处理中」时，编辑按钮灰显 |

**重要提示：** 即使存在数据依赖，用例**编写本身**仍可并行。
数据依赖只影响**测试执行顺序**，不影响用例**编写并行度**。
Writer Subagent 只需在前置条件中正确描述依赖关系即可。

---

## 拆分粒度建议

| 用例数量预估 | 建议拆分方式 |
|------------|------------|
| 总计 ≤ 20 条 | 单个 Writer，不拆分 |
| 20-50 条 | 按功能页面拆分（2-3个 Writer） |
| 50-100 条 | 按 CRUD 操作拆分（3-5个 Writer） |
| > 100 条 | 按 PRD 章节拆分（每章一个 Writer） |

---

## 拆分输出格式

分析完成后，向用户展示拆分方案，格式如下：

```
拆分方案：

模块：质量问题台账（菜单名）

┌─────────────────────────────────────────────────────┐
│ Writer A（可并行）: 列表页                           │
│   负责页面：质量问题台账 → 列表页                    │
│   功能子组：                                         │
│   - 搜索（P1: 按编号/状态/时间范围搜索）             │
│   - 列表展示（P0: 列表加载、列表列显示）             │
│   - 导出功能（P2: 导出 Excel）                       │
│   预计用例数: 12 条                                  │
├─────────────────────────────────────────────────────┤
│ Writer B（可并行）: 新增页                           │
│   负责页面：质量问题台账 → 新增页                    │
│   功能子组：                                         │
│   - 表单填写（P0: 正常新增流程）                     │
│   - 字段校验（P1: 必填/长度/格式校验）               │
│   - 字段联动（P1: 联动逻辑验证）                     │
│   预计用例数: 15 条                                  │
├─────────────────────────────────────────────────────┤
│ Writer C（可并行）: 详情/编辑页                      │
│   负责页面：质量问题台账 → 详情页、编辑页            │
│   功能子组：                                         │
│   - 详情查看（P1: 查看详细信息）                     │
│   - 编辑功能（P1: 字段修改）                         │
│   - 删除确认（P1: 删除校验和确认）                   │
│   预计用例数: 10 条                                  │
└─────────────────────────────────────────────────────┘

并行策略：A、B、C 三个 Writer 同时启动。
每个 Writer 输出时，module.name = "质量问题台账"，pages[].name = 各自负责的页面名。
Reviewer 合并时，将同名 module 下的 pages 合并。
```

## Writer 输出结构规则

每个 Writer 的 JSON 输出必须遵循以下层级：

```json
{
  "modules": [{
    "name": "质量问题台账",    // L2: 菜单/模块名（所有 Writer 使用同名）
    "pages": [{
      "name": "列表页",        // L3: 本 Writer 负责的页面
      "sub_groups": [{         // L4: 功能子组（可选）
        "name": "搜索",
        "test_cases": [...]
      }]
    }]
  }]
}
```

**关键规则：**
- 同一菜单下的多个 Writer，`modules[].name` 必须完全一致
- 每个 Writer 只负责自己的 pages，不跨页面
- Reviewer 负责合并同名 module 下的所有 pages
