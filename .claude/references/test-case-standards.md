# 测试用例编写标准

> 本文件是 FC/R/F 系列规则的单一权威源。所有 agent 通过引用本文件获取规则定义。
> 适用 agent：writer-agent (R01-R11)、reviewer-agent (F07-F15)、format-checker-agent (FC01-FC11)、standardize-agent

---

## 一、标题与结构规则

### R01 / FC01: 用例标题契约

- 中间 JSON 标题字段使用 Contract A：`title=验证xxx`，`priority` 字段独立填写（`P0`/`P1`/`P2`）
- Archive MD 展示标题组合为 `【P0】验证商品列表页默认加载`
- 正则校验：`^【P[012]】验证.+`
- 标题概括测试目的，不包含具体操作步骤

### R03 / FC03: 步骤编号

- 禁止步骤内容中出现编号前缀（`步骤1:`、`Step1:` 等）
- 正则检测：`^(步骤|Step)\s*\d+\s*[:：]`
- 编号由输出格式的 `step_number` 字段控制

## 二、步骤精度规则

### R02 / FC02: 首步格式

- 第一个步骤**必须**以「进入【完整导航路径】页面」开头
- 正则：`^进入【[^】]+】页面`
- 导航路径层级用 `→`（全角）连接，禁止 `->`、`>` 等
- 末尾必须包含等待条件

示例：
- 单层：`进入【商品列表】页面，等待列表数据加载完成`
- 多层：`进入【商品管理 → 商品列表】页面，等待表格至少显示 1 行数据或空态提示`

### R02 / FC07: 步骤三要素

每个步骤必须同时包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| 操作位置 | 控件所在区域 | 页面顶部搜索栏 / 表单区 / 表格操作列 / 弹窗内 |
| 操作对象 | 字段名 + 控件类型 | 「商品名称」输入框 / 「状态」下拉框 / 【提交】按钮 |
| 操作动作 + 具体值 | 动作 + 输入内容 | 输入"2026春季新款运动鞋" / 选择"上架" / 点击 |

典型违规：`搜索商品`（缺全部）、`点击搜索`（缺位置和对象）、`修改可编辑字段值`（缺具体字段和值）

### FC10: 异步等待条件

以下操作后**必须**注明等待条件：

| 操作类型 | 等待条件示例 |
|----------|-------------|
| 导航/跳转 | `等待列表数据加载完成` / `等待页面标题变为「xxx」` |
| 搜索/筛选 | `等待列表刷新完成` |
| 提交/保存 | `等待提交请求完成，页面跳转回列表页` |
| 弹窗操作 | `等待弹窗完全展开` |

## 三、内容质量规则

### R04 / FC04: 禁止模糊词（语义类）

| 模糊词 | 正则 | 替代方式 |
|--------|------|----------|
| 尝试 | `尝试` | 直接描述具体操作 |
| 相关 | `相关` | 列出具体字段/功能名 |
| 如/比如 | `[，,、]如[：:「]` 或 `比如` | 给出确切值 |
| 等 | `[，,]等[。，,]` 或 `等$` | 列出全部项目 |
| 某个/某些 | `某[个些]` | 指定具体对象 |
| 适当的 | `适当的` | 给出具体标准 |
| 正常的 | `正常的` | 描述具体表现 |

预期结果中的「正常」系列（「页面加载正常」「显示正常」「功能正常」）同样属于模糊词。

### R05 / FC05: 测试数据真实性

- 必须使用真实业务数据，给出具体输入值和选择项
- 禁止占位符：`test1`、`test123`、`abc`、`xxx`、`123456`、`asdf`、`样例数据`、`测试数据`
- 禁止不指定具体对象：「选择一个」「找到某个」「点击某个」

### R06 / FC06: 预期结果可断言化（语义类）

预期结果必须是 Playwright 可翻译为 `expect()` 断言的具体表现。

**禁止词：** `操作成功`、`显示正确`、`提交失败`、`加载正常`、`数据正确`、`功能正常`、`展示正确`、`信息正确`、`配置正确`

**正确写法：**
- 用可见文本：`页面提示"新增成功"` 而非 `操作成功`
- 用数据对比：`列表总数从"共 50 条"变为"共 51 条"` 而非 `列表更新`
- 用位置验证：`列表第一行「商品名称」列显示"2026春季新款运动鞋"` 而非 `新记录出现`
- 用元素状态：`【提交】按钮变为禁用状态（灰色不可点击）` 而非 `按钮不可用`

复合预期用分号分隔逐一列出。

### R10 / FC09: 前置条件操作化

前置条件须描述**如何达到该状态**，不能只说"需要什么状态"。

| 禁止写法 | 正确写法 |
|----------|----------|
| 系统中已有商品数据 | 已通过【商品管理→商品列表】页面新增商品"2026春季新款运动鞋"（分类: 运动鞋, 价格: 299.00） |
| 已登录系统 | 使用 admin 账号登录系统 |
| 已配置数据源 | 在【系统管理→数据源管理】页面已新增 Doris 数据源（host=10.0.0.1, port=9030） |
| 已创建 xxx | 给出完整创建步骤或 SQL |

**数据表场景额外要求：**
- 可执行 SQL 以 `DROP TABLE IF EXISTS` 开头
- 包含 `INSERT INTO` 样例数据（至少 1-3 条）
- 默认 Doris 语法，多数据源时各自给出

**多步表单必须闭合：** 涉及多步表单（Step 1 → Step 2 → ...）时，每步配置都须写完直到最终保存/提交。

## 四、格式规范规则

### R09 / FC08 / F09 / F11: 表单字段格式

同一表单多字段**必须**用 `\n` + `- 字段: 值` 列表格式，禁止堆在一行。

要求：
1. 必填字段前加 `*` 标记
2. 选项类字段直接给出值，不加冗余描述
3. 非必填字段不填时标注 `无`
4. 按页面视觉从上到下、从左到右排列
5. JSON `action` 字段中用 `\n` 分隔

正确格式：
```
点击【添加规则-完整性校验】，配置如下：
- *规则类型: 字段级
- *字段: info
- *统计函数: key范围校验
- 过滤条件: 无
- *校验方法: 包含
点击【保存】按钮
```

### R11 / FC11 / F12: 多项内容编号换行

- 前置条件、预期结果中 3 个及以上并列子项须用 `1)` `2)` 编号换行
- 预期结果禁止「或等价提示」「或类似说明」等模糊兜底

## 五、审查修正规则

### F07 / R07: 正向用例合并

- 同一 sub_group 内，`type: "positive"` 且步骤数 < 4 的用例应合并
- 合并后步骤按操作流顺序衔接，前置条件取并集
- **自动修正**：可自动合并

### F08 / R08: 逆向用例单一性

- 每条 `type: "negative"` 用例仅测试一个逆向条件
- 其余条件保持正确输入
- **自动修正**：多逆向条件拆分为多条独立用例

### F09: 表单字段合并（审查）

- 检测连续步骤中针对同一表单不同字段的独立输入操作
- **自动修正**：合并为一个字段列表步骤，按页面渲染顺序排列

### F10: 前置条件完整性

- 涉及数据表/同步/血缘的用例须有可执行 SQL
- **无法自动修正** → 标记 `[F10-MANUAL]`

### F11: 表单步骤换行（审查）

- 检测「依次填写」「分别输入」+ 逗号分隔的多字段操作
- **自动修正**：拆分为 `- 字段: 值` 列表格式

### F12: 多项内容编号换行（审查）

- 3 个及以上并列子项须编号换行
- **自动修正**：拆分为编号格式

### F13: 预期结果模糊兜底

- 正则：`或等[价效][的]?[提示说明表述]`、`或类似`
- **自动修正**：删除模糊兜底部分

### F14: 前置条件笼统概括

- 涉及文件/数据时须有具体内容，不接受「已准备好」等
- **无法自动修正** → 标记 `[F14-MANUAL]`

### F15: 前置条件多步操作闭合

- 多步表单须描述每步完整配置直至保存
- **无法自动修正** → 标记 `[F15-MANUAL]`

### 跨模块去重

- 标题编辑距离 < 30% 视为疑似重复
- 前置条件/页面不同则不视为重复
- 保留更完整的一条

## 六、质量门禁

### 问题率计算

```
问题率 = 含任意 F07-F15 问题的用例数 / 总用例数 × 100%
```

一条用例多个问题仅计 1 次。`[MANUAL]` 标记计入 `issues_found`，不计入 `issues_fixed`。

### 阈值决策

| 问题率 | 处理 | 输出 |
|--------|------|------|
| < 15% | 静默修正 | 修正后 JSON |
| 15-40% | 修正 + 警告 | 修正后 JSON + 警告信息 |
| > 40% | 阻断 | 问题报告，等待用户决策 |
