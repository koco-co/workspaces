# 测试用例编写规范

> 适用范围：所有自动生成的测试用例（Writer Subagent 和 Reviewer Subagent 必须遵守）

## XMind 层级结构

```
Root (项目名)
  └── L1: 需求名称（prd_id 写入 L1 labels）         ← trackerId 模块；版本已在 Root，L1 不加前缀
  └── L1: 【版本】需求名称（prd_id 写入 L1 labels）  ← 无 trackerId 模块；L1 带版本前缀
       └── L2: 菜单/模块名称          ← 如「商品管理」
            └── L3: 页面名称           ← 如「列表页」「新增页」「详情页」
                 └── [L4: 功能子组]    ← 如「搜索」「字段校验」（可选）
                      └── 用例标题
                           └── 步骤
                                └── 预期结果
```

- L1 格式由 config.json 中模块是否配置 `trackerId` 决定（见 xmind-structure-spec.md）
- L2 必须对应系统中的菜单或独立功能模块名称
- L3 按页面维度划分（列表页、新增页、编辑页、详情页等）
- L4 为可选层级，当页面内功能较多时按功能分组（搜索、导出、字段校验等）

## Archive MD 层级结构

Archive Markdown 从 `##` 开始，需求名称（`suite_name`）在 frontmatter 中，不再有 `#` 标题：

```
## 模块名称        ← 如「商品管理」（对应 XMind L2）
### 菜单名称       ← 如「商品列表」（对应 XMind L3）
#### 功能点名称    ← 如「搜索筛选」（对应 XMind L4，可选）
##### 【P0】验证xxxx  ← 用例标题（优先级前缀）
```

## 用例标题规范

- **优先级前缀**：`【P0】`/`【P1】`/`【P2】`，置于用例标题**前面**，如：`【P0】验证商品列表页默认加载`
- 「验证」二字紧跟优先级标记，如：`【P1】验证填写完整表单后成功提交`
- 标题应概括测试目的，不包含具体操作步骤

## 步骤格式规范

- **禁止**步骤编号前缀：不允许出现「步骤1:」「步骤2:」「Step1:」等字样
- 第一个步骤**必须**以「进入【完整导航路径】页面」开头
  - **单层菜单**：`进入【商品列表】页面`
  - **多层菜单**：`进入【商品管理 → 商品列表】页面`（菜单路径放在【】内，用 → 连接，导航路径**不放在括号注释中**）
  - **禁止格式**：`进入【商品列表】页面（导航路径：商品管理 → 商品列表）` ❌
- 步骤描述必须为可执行的具体操作，包含明确的操作对象和操作行为
- 步骤表格前**必须**有 `> 用例步骤` 标签

```markdown
> 用例步骤

| 编号 | 步骤 | 预期 |
| --- | --- | --- |
| 1 | 进入【商品管理 → 商品列表】页面 | 页面正常加载 |
```

## 用例设计原则

| 类型     | 条件组合规则           | 示例                                                                  |
| -------- | ---------------------- | --------------------------------------------------------------------- |
| 正常用例 | 可包含多个正向条件组合 | 验证填写完整表单后成功提交（商品名称 + 商品分类 + 价格 + 库存数量） |
| 异常用例 | **仅允许一个逆向条件** | 验证商品名称为空时无法提交                                          |
| 边界用例 | 针对边界值的单一验证   | 验证商品名称输入 100 个字符时正常保存                               |

### 正向用例合并规则（强制）

同一功能区（sub_group 或 page）内，步骤数少于 4 条的正向用例必须与其他正向用例合并：

- 将该功能区内所有步骤数 < 4 的正向用例整合为 1 个综合正向用例
- 综合用例步骤按用户操作流顺序衔接，前置条件取各子用例的并集
- **异常用例不适用此规则**：每条异常用例仍只测试一个逆向条件，步骤数即使只有 1 条也不与其他用例合并

## 测试数据规范

- **必须使用真实业务数据**，给出具体的输入值和选择项
- **禁止模糊词**：

  | 禁止词                     | 正确写法                                                         |
  | -------------------------- | ---------------------------------------------------------------- |
  | 尝试提交表单               | 点击【提交】按钮                                                 |
  | 输入像"测试数据"这样的内容 | 在「商品名称」输入框输入"2026春季新款运动鞋"                    |
  | 选择如"已完成"等状态       | 在「状态」下拉框选择"上架"                                     |
  | 输入比如手机号             | 在「联系电话」输入框输入"13800138000"                            |
  | 填写相关信息               | 在「商品名称」输入"2026春季新款运动鞋"，在「商品分类」选择"运动鞋" |

## 预期结果规范

- 必须精确描述系统行为，包含可观测的具体表现
- **禁止空洞表述**：

  | 禁止写法 | 正确写法                                                                         |
  | -------- | -------------------------------------------------------------------------------- |
| 操作成功 | 页面提示"新增成功"，列表页新增一条记录，商品名称显示为"2026春季新款运动鞋"           |
| 显示正确 | 列表第一行显示：商品名称"2026春季新款运动鞋"，商品分类"运动鞋"，状态"上架"          |
| 提交失败 | 「商品名称」输入框下方显示红色提示"请输入商品名称"，【提交】按钮保持不可点击状态 |

## 前置条件数据准备规则（数据表 / 血缘 / 同步 / 回刷场景）

**当需求涉及数据表操作、数据接入、批量同步、任务调度、对账、规则配置、血缘解析、指标生成、历史升级/补数/回刷等能力时，必须给出具体的可执行前置条件，与 config.repos 是否为空无关。**

格式示例：

```text
1、环境说明：目标模块已部署，数据源连接正常

2、Doris2.x SQL语句准备:
DROP TABLE IF EXISTS qa_test.table_a;
CREATE TABLE qa_test.table_a (
  id INT,
  name VARCHAR(100),
  value DECIMAL(10,2),
  status VARCHAR(20),
  create_time DATETIME
);
INSERT INTO qa_test.table_a VALUES
  (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');

3、Hive2.x SQL语句准备:
CREATE TABLE IF NOT EXISTS qa_test.table_a (
  id INT,
  name STRING,
  value DECIMAL(10,2),
  status STRING,
  create_time TIMESTAMP
);
INSERT INTO qa_test.table_a VALUES (1, '测试数据A', 100.00, '启用', '2026-01-01 10:00:00');
```

- 表名和字段名必须与 PRD 业务场景一致，不得用 `test1`、`abc` 等占位符
- 多数据源时每种数据源单独给出（Doris 用 `VARCHAR`，Hive 用 `STRING` 等）
- 只列 PRD 中实际涉及的数据源，无需穷举

### 血缘 / 同步 / 回刷类需求的额外硬约束（强制）

当 PRD 中出现以下任一关键词时，前置条件不能只写建表和 INSERT，还必须写出“链路如何建立”：

- `血缘`、`上下游`、`解析`
- `脚本模式指标`、`向导模式指标`
- `同步`、`调度任务`
- `历史数据`、`升级`、`补数`、`回刷`

此类场景的前置条件至少包含：

1. **执行目标**：在哪个引擎 / 集群执行；若 Doris2.x、Doris3.x 复用相同库表名，必须明确“本步骤在哪个集群执行”
2. **建表 / 灌数 SQL**：真实表结构与真实业务数据
3. **建链 SQL / 任务 SQL**：如 `INSERT OVERWRITE`、调度任务 SQL、解析链路所依赖的实际 SQL，不能只写“确保建立血缘关系”
4. **指标定义 / 脚本 SQL**：若需求涉及指标，必须写出原子/派生/复合指标来源，或给出脚本模式完整 SQL
5. **历史回刷 SQL**：若验收点涉及历史兼容、升级后可用，必须给出历史样例准备方式或回刷 SQL
6. **执行顺序**：若结果依赖“先建链再同步”“先下线再刷新”，必须写明顺序

**禁止写法：**

- `确保它们与 xxx 建立可追溯血缘关系`
- `保留查询结果的上游表记录`
- `准备历史指标样例`
- `同步相关数据后即可验证`

**正确写法：**

```text
4、在 Doris2.x 集群执行建链 SQL：
INSERT OVERWRITE TABLE qa_test.lineage_mid
SELECT ... FROM qa_test.lineage_src;

INSERT OVERWRITE TABLE qa_test.lineage_dwd
SELECT ... FROM qa_test.lineage_mid;

5、在指标平台创建脚本模式指标，SQL 如下：
SELECT SUM(dwd.amount)
FROM qa_test.lineage_dwd dwd
JOIN qa_test.lineage_mid mid ON dwd.id = mid.id;

6、执行历史回刷 SQL：
SELECT dt, SUM(amount) AS history_amount
FROM qa_test.lineage_dwd
GROUP BY dt;
```

## 源码优先规则（当 config.repos 非空时启用）

> 以下规则仅在 config.json 中 `repos` 字段为非空对象时适用。若 `repos: {}` 则跳过本节。

- **源码优先**：用例编写前必须先确认 `.repos/` 中相关仓库已切到 config.json 的 `branchMapping` 字段解析出的目标分支。
- **步骤格式**：表单类步骤优先写成结构化块，如：

```text
点击【新增商品】按钮，基础信息如下：
- 商品名称: 2026春季新款运动鞋
- 商品分类: 运动鞋
- 价格: 299.00
```

- **禁止”堆句子”**：不得将多个字段输入动作压缩为”填写相关信息””勾选对应字段”这类概括句。

## 表单多字段步骤合并规则（强制）

**当一个步骤涉及同一表单/面板内的多个配置项时，必须合并为一个步骤节点，按字段在页面上的实际顺序逐项列出。**

**正确写法：**

```text
在【新增商品】表单页面，点击【保存】前按顺序配置如下：
- 商品名称: 2026春季新款运动鞋
- 商品分类: 运动鞋
- SKU编号: SKU-2026-001
- 价格: 299.00
- 库存数量: 100
- 状态: 上架
- 商品卖点: 轻量缓震跑鞋（非必填，可填写无）
```

**禁止写法（每个字段单独一个步骤节点）：**

```text
❌ 步骤1: 在「商品分类」下拉框选择「运动鞋」
❌ 步骤2: 在「商品名称」输入框输入「2026春季新款运动鞋」
❌ 步骤3: 在「价格」输入框输入「299.00」
```

**适用规则：**

1. **字段顺序**：按前端页面实际渲染顺序排列，优先参考前端源码（表单组件定义顺序）
2. **非必填字段**：保留在列表中，在值后加注 `（非必填，可填写无）`
3. **嵌套配置**：若某个字段展开后有子配置项，用缩进或换行继续列举（如「计算逻辑配置」下有「对比方法」）
4. **多步骤拆分时机**：仅在以下情况允许拆分为单独步骤：
   - 该字段的选择会触发页面联动/刷新，且联动后需要单独验证
   - 该步骤本身就是该用例的核心验证动作（如验证某个字段选项是否存在）

## 质量控制

Reviewer 采用 3 级质量阈值自动决策：

| 问题率    | 行为                                 |
| --------- | ------------------------------------ |
| < 15%     | 自动修正，无需通知用户               |
| 15% - 40% | 自动修正 + 质量警告，建议用户核查    |
| > 40%     | 立即停止，输出阻断报告，等待用户决策 |

问题率 = 含 F01-F18 任意问题的用例数 / 总用例数（一条用例多个问题只计 1 次）。
