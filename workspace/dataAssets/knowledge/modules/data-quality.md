# 数据质量模块业务规则

数据质量模块按功能分为以下子模块，业务流程存在前后依赖关系：**规则库配置 → 规则集管理 → 规则任务管理 → 校验结果查询 → 数据质量报告**。

## 1. 规则库配置

- 每条规则都有**状态**字段，必须设为「开启」
- 只有开启状态的规则才会在「规则集管理 → 监控规则 → 添加规则」的下拉选项中显示
- 未开启的规则不会出现在下游引用入口

## 2. 规则集管理

- 选择库表：支持的数据源类型 = **sparkthrift / doris / hive**
- 容量上限：最多 **20 个规则包 × 每包 10 条监控规则**
- 规则集保存后，规则任务管理才能引用其规则包
- 规则集 = 「库表 + 规则包集合」的可复用配置单元

## 3. 规则任务管理

- 选择库表时，必须**与规则集管理中已存在的记录保持一致**（同数据源 + 同库 + 同表）
- 创建流程：Step1 关联库表 → Step2 引入规则集的规则包 → Step3 设置调度属性 → 保存
- **执行入口**：点击任务行的**表名**展开抽屉，在抽屉中点击【立即执行】
  - 非「执行」按钮；列表行上没有直接的执行按钮
- 执行后，会在**校验结果查询**中生成一条新记录

## 4. 校验结果查询

- 每次任务执行 → 生成一条记录
- 记录状态语义：
  - **校验通过**：所有规则均通过
  - **校验异常**：任一规则不通过 **或** 任务运行失败（如 SQL 错误、引擎异常）
- 抽屉详情：点击记录的表名 → 展开详情抽屉 → 显示【校验未通过(N)】统计 + 有效性校验规则卡片
- 规则卡片操作链接：**【查看明细】**、**【查看趋势】**（非「查看详情」）

## 5. 数据质量报告

- 选择库表：该库表**必须在校验结果查询中已存在记录**（即任务已执行过）
- 报告周期默认使用「一次性」，除非用例明确要求其他周期
- 点击确定后，**等待约 15 分钟**，已生成报告列表中才会出现新记录
- 报告生成是异步的，需轮询

## 6. 通用配置 → 报告关联维表设置

- 默认**不进行**维表关联配置
- 仅当需求/用例明确要求产生「车辆数据」相关报告时才配置
- 若配置不匹配，规则集/任务创建向导中会出现「是否需要车辆信息」类选项

## 7. json 格式校验管理

- 仅针对两类场景做正则配置：
  - **完整性校验 → key 值的存在性**
  - **有效性校验 → key 值对应的 value 正确性**
- 不涵盖其他字段级校验场景

## 8. 项目管理

- 在数据质量中做任何配置前，必须**先新建一个项目**
- qa-flow 自动化默认使用项目 **`pw_test`**（pw 代表 playwright）
- QUALITY_PROJECT_ID 会因环境重建而变化，脚本应动态查询而非硬编码

## 9. 适用范围与客户约定

### 9.1 岚图汽车专属

- 本 module 描述的全部业务规则**仅适用于岚图汽车项目**（源码仓库位于 `customltem/`）
- **标品项目不适用**，但可作为理解数据质量模块通用语义的参考
- 岚图专属字段（如「车辆信息」维表关联）、容量上限（20 × 10）、交互（点表名展开抽屉→立即执行）都可能在标品中不一致

### 9.2 数据源优先级（写用例时的选型顺序）

写 Archive 用例和自动化脚本时，数据源选型按以下顺序：

1. **sparkthrift2.x**（首选）
2. **doris3.x**（次选）
3. **hive2.x**（兜底）

### 9.3 前置条件 SQL 规范

- 用例的前置条件必须**明确列出建表 + 初始化数据的 SQL**，按所选数据源的方言写
- **表名可以在不同数据源间保持一致**（如 `quality_test_num`），方便用例步骤中直接引用而无需区分数据源
- 多数据源矩阵跑同一用例时（如 sparkthrift + doris），前置条件应提供每种数据源的建表 SQL，表名相同、字段类型按方言调整
- 建表语句一般需包含 `DROP TABLE IF EXISTS` + `CREATE TABLE` + `INSERT` 三段，保证可重入
